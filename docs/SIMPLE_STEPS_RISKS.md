# Анализ Рисков: Простые Шаги Улучшения Оркестрации

## Шаг Б: Быстрый откат при падении Quality Gate

**Что было сделано:**
В `ClaimService.java` в методе `complete` добавлена логика быстрого возврата задачи при падении проверок:
```java
// Implementer finished, move to review stage
task.setStatus(TaskStatus.review);
gateOrchestrator.runQualityGate(task);
if (!task.isQualityGatePassed()) {
    task.setStatus(TaskStatus.queued);
}
```

**Цель:**
Разгрузить AI-агентов, выступающих в роли REVIEWER. Если автоматические проверки (Quality Gate) не пройдены, нет смысла отправлять задачу на ручное или AI ревью. Задача сразу возвращается в очередь (`queued`) для переделки.

**Риски и возможные последствия:**
1.  **Бесконечный цикл (Infinite Loop):** Если проверки в `GateOrchestrator` "флакают" (flaky tests) или их требования невыполнимы для агента-исполнителя, задача попадет в бесконечный цикл `queued` -> `claimed` -> `complete` -> `fail quality gate` -> `queued`. Агенты будут бесконечно тратить ресурсы API, пытаясь выполнить задачу, которую невозможно сдать.
2.  **Потеря обратной связи:** Раньше REVIEWER мог дать человекочитаемый фидбек агенту, почему задача не принята. Теперь агент-исполнитель должен быть способен сам прочитать и понять структуру JSON в поле `quality_gate_report`, чтобы при следующей итерации исправить ошибки. Если промпт агента не включает этот репорт при повторном `dispatch`, он просто сделает то же самое.

**Как контролировать:**
Необходимо добавить механизм счетчика попыток (retry limit) или экстренного эскалирования задачи (например, перевод в статус `blocked`), если она провалила Quality Gate 3 раза подряд. Также следует убедиться, что `JulesDispatchService` прикладывает результаты `quality_gate_report` к промпту при повторном взятии задачи в работу.
