# Анализ Рисков: Улучшения Оркестрации

## Шаг Б: Быстрый откат при падении Quality Gate (Итерация 3)

**Что было сделано:**
1. В `ClaimService.java` в методе `complete` добавлена логика быстрого возврата задачи при падении автоматических проверок (Quality Gate).
   - Если `gateOrchestrator.runQualityGate(task)` помечает `qualityGatePassed` как `false`, задача не отправляется на этап `review`.
   - Текущий `ClaimEntity` корректно помечается как `failed` и закрывается.
   - Аккаунт исполнителя освобождается (`idle`).
   - Поле `retry_count` (добавленное в V27) инкрементируется.
   - Если `retry_count` достигает 3, задача переводится в статус `blocked`. Иначе возвращается в `queued`.

2. Внедрен механизм сохранения аудиторского следа (Audit Trail):
   - Создана таблица `task_gate_logs` (миграция V28).
   - `GateOrchestrator` теперь сохраняет каждый результат запуска проверок в новую сущность `TaskGateLogEntity`. Это решает проблему перезаписи JSON в поле задачи.

**Цель:**
Разгрузить AI-агентов (роль REVIEWER) от проверки заведомо нерабочего кода, корректно управлять жизненным циклом Claims для предотвращения утечек аккаунтов, обеспечить лимиты на переделки и сохранить полную историю проверок для аналитики.

**Решённые риски (из предыдущих аудитов):**
1.  **Бесконечный цикл (Infinite Loop):** Решено внедрением `retry_count` и статуса `blocked`.
2.  **Утечка аккаунтов (Account Leak):** Решено корректной отработкой `ClaimEntity` (переход в `failed`) и освобождением аккаунта при провале гейта.
3.  **Потеря аудиторского следа:** Решено созданием `TaskGateLogEntity`. Теперь мы имеем исторические логи для каждого падения задачи, независимо от того, как часто она запускается.

**Оставшиеся системные моменты:**
1.  **Разделение причин провала (Flaky vs Code Error):** Механизм всё ещё считает любые падения Quality Gate как +1 к `retry_count`.
2.  **Передача контекста агенту:** `JulesDispatchService` должен быть настроен так, чтобы извлекать последний `quality_gate_report` из задачи и добавлять его в промпт при повторной диспетчеризации (когда `retry_count > 0`), чтобы агент понимал, что конкретно нужно исправить. На данный момент мы сохранили отчет, но не скормили его обратно агенту в промпте.
