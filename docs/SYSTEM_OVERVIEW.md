# ОБЩИЙ СИСТЕМНЫЙ И БИЗНЕС-ОБЗОР: EneikProductionSys
**Версия:** 2026.07 | **Статус:** Единый источник правды (Single Source of Truth)

---

## 1. Введение и бизнес-функции системы

**EneikProductionSys** — это передовая экосистема автоматизированного инжиниринга программного обеспечения, построенная на базе ролевой агентной оркестрации. Она имитирует полный цикл разработки программного продукта (от декомпозиции сырых требований пользователя до проектирования, написания кода, контроля качества, авто-мержа и деплоя) силами 12 специализированных виртуальных агентов (ролей `BARCAN-TAG-00` — `BARCAN-TAG-11`).

### Основные бизнес-процессы (End-to-End)
1. **Декомпозиция требований (Requirement Decomposition):** Бизнес-пользователь отправляет сырые требования на эндпоинт `/api/requirements`. Система использует `DecompositionService`, чтобы разбить требования на атомарные задачи, автоматически назначив им нужные теги ролей.
2. **Оркестрация очереди и Claim-логика (Claim Orchestration):** Задачи попадают в глобальную очередь в статусе `queued`. Агенты (виртуальные аккаунты разработчиков/тестировщиков) вытягивают задачи из очереди, вызывая метод `POST /api/tasks/claim`. Во избежание race conditions используется конкурентная блокировка БД `FOR UPDATE SKIP LOCKED`.
3. **Гейты качества (Quality Gates):** Выполнив задачу, агент отправляет её на проверку. Система переводит задачу в статус `review` и пропускает её через серию автоматизированных гейтов (проверка бизнес-ценности, наличие тестов, версионная совместимость API, адаптивность скриншотов).
4. **Защита потока (Lease Watchdog):** Специальная служба `LeaseWatchdogService` каждые 60 секунд проверяет активные задачи. Если агент взял задачу в работу (статус `claimed`), но пропал из сети или затянул её выполнение (истек таймаут аренды lease), Watchdog автоматически возвращает задачу в статус `queued`, помечаетclaim как `expired`, а аккаунт агента переводит в `offline`.
5. **Ежедневный цикл самоанализа (Self-Falsification Cycles):** Каждую ночь система запускает проверку кодовой базы на соответствие жестким критериям отказа (Refusal Criteria) каждой роли. При обнаружении регрессии качества автоматически генерируются новые задачи на исправление кода.

## 2. Архитектура и сетевой ландшафт

Система состоит из трех независимых взаимодействующих сервисов, работающих в едином контуре Docker Compose:

```text
       +--------------------------------------------+
       |             Svelte Frontend                |
       |               (Порт 3000)                  |
       +---------------------+----------------------+
                             | REST API
                             v
       +---------------------+----------------------+
       |            Spring Boot Backend             |  <---+ Периодический Watchdog
       |               (Порт 8080)                  |
       +----------+----------------------+----------+
                  |                      |
                  | REST (AI review)     | JPA/Hibernate
                  v                      v
       +----------+----------+     +-----+----------+
       |  Python FastAPI ML  |     |   Database     |
       |    (Порт 8000)      |     | (H2 / Postgre) |
       +---------------------+     +----------------+
```

1. **Spring Boot Backend (Java 17, порт 8080):** Ядро системы. Управляет бизнес-логикой, жизненным циклом задач, транзакциями, хранением состояния и шедулерами. Использованы Flyway-миграции (V1 — V29).
2. **Svelte UI Frontend (порт 3000):** Панель администратора (Admin Dashboard). Позволяет визуализировать очередь задач, статус и загрузку ролевых агентов, список проектов, лог гейтов качества и возникшие бутылочные горлышки (bottlenecks).
3. **FastAPI ML Service (Python 3, порт 8000):** Аналитический контур. Предоставляет математический аппарат для прогнозирования бутылочных горлышек на основе WIP-лимитов и времени цикла, а также API для LLM-анализа соответствия PR правилам ролей.

## 3. Модель жизненного цикла задачи (Task Lifecycle)

Каждая задача в системе проходит строгую цепочку состояний:

```text
    [ queued ]
        |
        |  POST /api/tasks/claim  (Конкурентная выборка FOR UPDATE SKIP LOCKED)
        v
    [ claimed ] <---+
        |           | Lease Watchdog (раз в 60 сек сбрасывает зависшие задачи)
        |           +--------------------------------------------------------+
        |  claimService.complete(taskId)                                     |
        v                                                                    |
    [ review ]                                                               |
        |                                                                    |
        +---> [ Гейты качества: Business Value, Tech, Design, Compliance ] ----+
        |                                                                    |
        | Все гейты пройдены успешно                                         |
        v                                                                    |
    [  done  ]                                                               |
                                                                             |
    [ failed ] <-------------------------------------------------------------+
              Отказ в прохождении гейта или критический сбой
```

### Детали переходов:
* **queued:** Задача создана и ожидает претендента.
* **claimed:** Задача захвачена агентом. Аккаунт агента переводится в статус `busy` (занят). Поля `claim_id` and `lease_expires_at` заполняются.
* **review:** Агент завершил работу. Запускается контур проверок. Для автоматического ревью может подключаться AI-сервис.
* **done:** Успешное слияние кода в `main` (авто-мерж) и фиксация бизнес-ценности.
* **failed:** Задача заблокирована из-за критической ошибки, нарушающей критерии приемки, или не пройдена верификация безопасности.

## 4. Единая карта 12 ролей (BARCAN-TAG) и их группы

Все роли сеются в БД при запуске системы через миграцию `V3__core_agency_schema.sql` и описываются индивидуальными файлами `BARCAN-TAG-XX.md` в корне репозитория.

### Группа A — Проектирование, смысл и границы
* **TAG-00 (CODE-GUARDIAN):** Tech Lead / Code Reviewer. Осуществляет проверку кода, контролирует цикломатическую сложность и предотвращает разрастание God-files (>50КБ).
* **TAG-01 (ACTUALIST-OBJECT):** Solution Architect. Проектирует сущности данных, выстраивает границы доменов и строго разделяет слои MVC (например, блокирует прямые SQL-запросы в контроллерах).
* **TAG-02 (RIGID-DESIGNATOR):** Backend Engineer. Разрабатывает API эндпоинты, реализует бизнес-логику и обеспечивает покрытие unit-тестами.
* **TAG-09 (MORAL-DILEMMA):** Technical Product Manager / Mediator. Обладает монопольным правом на запуск декомпозиции и компиляцию задач из вишлиста проекта. Отвечает за баланс скорости и качества.

### Группа B — Данные, логика и вычисления
* **TAG-04 (MODAL-QUANTIFIER):** ML Engineer. Разрабатывает модели для анализа и прогнозирования бутылочных горлышек в цепочке TOC.
* **TAG-08 (SUBSTITUTIVITY-SALVA-VERITATE):** Data Engineer / DBA. Контролирует схемы реляционных БД, Flyway-миграции данных, оптимизирует запросы.

### Группа C — Человеческий фактор и восприятие
* **TAG-03 (BELIEF-INTENSION):** UI/UX Designer. Проектирует визуальные интерфейсы, спецификации дизайн-системы и макеты экранов.
* **TAG-11 (CLIENT-PERCEPTION):** Frontend Engineer. Реализует клиентский интерфейс на Svelte, интегрирует бэкенд API и контролирует производительность рендеринга.

### Группа D — Надежность, верификация и безопасность
* **TAG-05 (NECESSARY-IDENTITY):** SRE / DevOps. Настраивает CI/CD-пайплайны, предотвращает утечку секретов и контролирует чистоту репозитория от мусора.
* **TAG-06 (DEONTIC-CONSISTENCY):** QA Automation. Пишет сквозные интеграционные и E2E автотесты, проверяя систему на наличие дефектов.
* **TAG-07 (SECOND-ORDER-KNOWLEDGE):** AppSec / Security. Обеспечивает безопасность, авторизацию и защиту персональных данных.

### Группа E — Регуляция и комплаенс
* **TAG-10 (DEONTIC-PROHIBITION):** Compliance Officer. Контролирует соответствие требованиям безопасности данных, накладывает маскирование персональных данных (PII).

## 5. Разделение файлов документации: Рантайм-активные vs. Справочные

Для стабильности работы системы все файлы четко разделены по назначению.

### 5.1. Рантайм-активные ресурсы (Используются кодом)
Любые перемещения или удаления этих файлов приведут к сбоям в работе бэкенда или ML-сервиса:
1. `/BARCAN-TAG-XX_*.md` (в корне проекта) — загружаются `RoleCapabilityLoader` и парсятся для анализа критериев отказа.
2. `docs/DESIGN_SYSTEM.md` — проверяется в `TechnicalLeadCompiler` при верификации UI-задач.
3. `docs/AI_REVIEW_GUIDELINES.md` — используется бэкендом для настройки промптов внешнего ИИ-ревьюера.

### 5.2. Справочные ресурсы (Чисто информационная документация)
Эти файлы служат справочными материалами для разработчиков и не влияют на выполнение кода:
1. `docs/SYSTEM_OVERVIEW.md` — данный обзорный мастер-документ.
2. `docs/DOCUMENTATION_ROADMAP.md` — план реорганизации и очистки легаси-файлов.
3. `docs/DIRECTORY_MAP.md` — карта разделения ответственности по каталогам.
4. `docs/SEQUENCE_FLOW.md` — диаграммы сетевых вызовов.
5. `docs/SCHEMA_CONTRACT.md` — структура базы данных.

## 6. Зоны архитектурного риска

Аудит системы вскрыл важные "бутылочные горлышки" в контуре контроля качества:

### 6.1. Уязвимость Fail-Open (Аварийное открытие)
В случае, когда сервис машинного обучения (FastAPI @ 8000) выключен, падает под нагрузкой или испытывает проблемы с сетью, контур верификации PR (`JulesDispatchService` и `MLPredictionServiceClient`) не блокирует процесс, а применяет стратегию **Fail-Open**:
* Ошибка сети молча подавляется в блоке `catch`.
* Клиент возвращает `approved = true`.
* PR автоматически сливается в основную ветку с шуточным комментарием: `"Antigravity fallback review pass."`.

**Риск:** Это открывает вектор для незаметного прохождения дефектного или небезопасного кода при временном сбое инфраструктуры.

### 6.2. Рекомендации по переходу к Fail-Safe (Аварийное закрытие)
Для устранения этой уязвимости необходимо:
1. При падении AI-сервиса возвращать `approved = false` с кодом `VERIFICATION_SERVICE_UNAVAILABLE`.
2. Блокировать автоматическое слияние в ветку `main`.
3. Переводить PR и задачу в специальный статус **`needs_human_review`** (требуется ручная проверка техлидом).
4. Устанавливать жесткий SLA (4–12 часов), по истечении которого в Linear отправляется критический алерт, а приоритет задачи в БД повышается до максимального уровня `1000`.
