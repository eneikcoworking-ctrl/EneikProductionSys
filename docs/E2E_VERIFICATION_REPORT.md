### Найденные проблемы

1.  **Критическая: Отсутствие Watchdog**. В коде backend (Spring Boot) не обнаружено логики по расписанию (`@Scheduled`) или фоновых процессов, которые бы проверяли `claims.lease_expires_at`. Это блокирует самовосстановление системы при падении агентов.
2.  **Блокирующая: Ошибка компиляции в `TaskRepository.java`**. Метод `countByStatus` был продублирован, что предотвращало сборку проекта. (Исправлено в рамках данной сессии).
3.  **Edge Case: Отсутствие Account API**. Создание аккаунтов возможно только через прямые манипуляции с БД, что затрудняет интеграцию новых агентов.
4.  **H2 Locking**: Файл базы данных блокируется процессом Spring Boot, что делает невозможным использование стандартных инструментов `RunScript` (H2 CLI) паралллельно с работающим приложением без перевода БД в серверный режим.

### Что проверено, но не входило в основной сценарий
-   **Heartbeat API**: Эндпоинт `POST /api/tasks/{id}/heartbeat` существует и обновляет `lease_expires_at`, но в данном прогоне его влияние на Watchdog не могло быть проверено из-за отсутствия самого Watchdog.
-   **ML Service**: Интеграция с ML-сервисом для предикции бутылочного горлышка проверена через `/api/dashboard/bottlenecks` — запросы уходят, ML-сервис (Python) отвечает корректно.

### Рекомендация по следующему шагу
**Реализовать фоновый процесс Watchdog.**
Необходимо добавить в `ClaimService` или отдельный `LeaseWatchdogService` метод с аннотацией `@Scheduled(fixedRate = 60000)`, который будет выполнять:
```sql
UPDATE tasks SET status = 'queued' WHERE id IN (SELECT task_id FROM claims WHERE released_at IS NULL AND lease_expires_at < NOW());
UPDATE claims SET result_status = 'expired', released_at = NOW() WHERE released_at IS NULL AND lease_expires_at < NOW();
```
И переводить соответствующих агентов в `status = 'offline'`.
