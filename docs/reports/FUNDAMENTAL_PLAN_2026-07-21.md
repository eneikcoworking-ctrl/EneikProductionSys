# Фундаментальный план правок — версия 2 (серьёзный проход, 24ч на исправление)

Версия 1 (ниже, раздел "История") содержала P0-фиксы, написанные под давлением в течение ночи. При построчной проверке (следующий проход, "опиши флоу и найди недочёты") выяснилось, что часть из них либо не работает для мотивирующего кейса, либо основана на неверной посылке. Эта версия — результат честной, построчной перепроверки каждого фикса перед тем как продолжать. У нас есть сутки (лимиты Jules сбросятся) — план рассчитан на осторожную, тестируемую реализацию, а не на очередной рывок.

## 0. Ф7/Ф8 — архитектурная переработка эпиков (операторская директива, после версии 1)

Оператор указал на две дополнительные вещи по ходу разбора:

**Ф7 — account-пиннинг у feature-thread continuation убран.** `JulesDispatchService.dispatchInternal` требовал совпадения аккаунта с тем, что аутентифицировал исходную ветку, прежде чем разрешить продолжение работы на той же ветке. Проверено: ничего технически это не обосновывает (авторизация запроса создания сессии — только по API-ключу для квоты/биллинга, доступ к репозиторию — на уровне коллабораторов; branch protection на этих репозиториях вообще не активна). Комментарий в коде сам признавался "never been tested". Хуже — `AccountRepository.lockNextJulesAccountWithCapacity` не знает о `featureId` вообще, так что ограничение никогда и не приносило пользы (continuation срабатывал только по счастливой случайности раунд-робина). Убрано полностью, не заменено на "thread-aware диспетчер" — раз пользы не было, чинить нечего.

**Ф8 — модель "1 вишлист = 1 эпик" заменена на "1 вишлист = N эпиков".** Раньше `FeatureEntity` был голой группирующей строкой (id/projectId/rootWishlistId), а JTBD/Kano/Cynefin целиком жили на уровне таска. Теперь:
- Вишлист сначала разбивается на N эпиков по нарративам/темам (новый первый проход компиляции) — не всегда ровно один.
- Каждая последующая компиляция (не только первая) явно решает по каждому эпику: совпадает семантически с уже существующим эпиком проекта (ИИ сравнивает по списку id+title+jtbd, переданному в промпт) или это новый эпик.
- Контент: эпик = JTBD(для заказчика) + Kano + Cynefin + sixSigmaMetric + tocConstraintRef (агрегированные, бизнес-уровня). Задача = JTBD(для эпика, не для заказчика) + Cynefin + своя sixSigmaMetric/tocConstraintRef (технические, узкие). Kano — только на уровне эпика, leanValue — только на уровне задачи (операторские решения).

**Реализация**: миграция V49 (`features` таблица получила title/jtbd/kano_class/cynefin_domain/six_sigma_metric/toc_constraint_ref), новый record `MLPredictionServiceClient.EpicPlan` (оборачивает список `TaskSliceMetadata`, который потерял `kanoClass`/`sourceIndex`), промпт компилятора переписан на двухуровневую схему с явным списком существующих эпиков, `parseCompilerPlan`/`isValidCompilerPlan`/`buildTaskGraphFromSlices` переписаны под `List<EpicPlan>`, `FeatureService` получил `listExistingEpics`/`findExistingEpic`/`createFeature`.

**Не тронуто в этом проходе, задокументировано честно**: `TechnicalLeadCompiler`'s собственная независимая text-based эвристика Kano (`kanoClass(wishlist, extractedRoles)`, используется для EMS-метрик `kpiWeight`/`criticalityScore`) и `CommandDashboardService.classifyKano` (ключевая-слова эвристика для recommendation-текста) продолжают работать как раньше — НЕ читают новое структурированное `FeatureEntity.kanoClass` вообще. Ничего не ломается (обе эвристики самодостаточны), но реального выигрыша от новых структурированных данных эпика эти два потребителя пока не получают — отдельный, более крупный follow-up, не блокирующий сегодняшнюю работу.

**Тесты**: `EpicDecompositionIntegrationTest` (3 кейса: один вишлист → несколько разных эпиков; повторная компиляция переиспользует существующий эпик по `existingEpicId`; галлюцинированный `existingEpicId` откатывается на создание нового эпика вместо падения). Плюс исправлены 7 существующих тестов в `AutonomousPipelineIntegrationTest`/`ProjectFlowIntegrationTest`, использовавших старую сигнатуру.

## 1. Проверенные находки (с доказательствами, не с предположениями)

### Ф1 — `.eneik/*` бэкенд-резолюция конфликтов (P0.1, ночная версия) физически не работает для мотивирующего кейса — ПОДТВЕРЖДЕНО
Прочитал `fetchPrFiles`: `GET /pulls/{n}/files` возвращает **все** файлы диффа PR, не только конфликтующие. У PR#12 в диффе одновременно `.eneik/task-plan.json` и весь реальный Svelte-код. Проверка `files.allMatch(f -> f.startsWith(".eneik/"))` для такого PR всегда `false` — фикс не срабатывает именно там, где обещал сработать. Работает только для PR, где абсолютно все файлы под `.eneik/` (то есть только для системных record-PR, где конфликт и так дёшев).

### Ф2 — гейт на компиляцию нового вишлиста "деадлочит" recovery-путь — ОПРОВЕРГНУТО, снято
Проверил: `role_mismatch_followup` вишлисты (эскалация конфликта) идут через отдельный `compilePendingRecoveryWishlist → tryCompileWishlistCheaply`, который не проходит через `dispatchBatchedWishlistCompiler` вообще ("дешёвый детерминированный путь", без ИИ). Мой гейт эту ветку не видит. Никакого дедлока нет. Ошибочная находка предыдущего прохода, снимаю её.

### Ф3 — `dependsOn`-гейт проверяет `TaskStatus.done`, а не факт мержа — ПОДТВЕРЖДЕНО
Собственный комментарий `ClientDeliverableReadinessService`: `done` ставится при апруве ревью, **независимо от того, смержился ли PR**. `dispatchQueuedTasks`'ин чек `dependsOn.getStatus() != TaskStatus.done` пропустит следующую стадию, даже если PR зависимости всё ещё висит несмерженным (конфликт/CI). Стадия B может стартовать не видя реального кода стадии A в main — именно то, что фикс должен был исключить.

### Ф4 — `dependsOn`, указывающий на заброшенный (эскалированный) таск, вешает даунстрим навсегда — ПОДТВЕРЖДЕНО
Recovery-эскалация (`AutoMergeService`, 3 попытки исчерпаны) создаёт **новый** taskId вместо починки старого; старый помечается `failed` и никогда не станет `done`. Ничего в кодовой базе не ищет и не перепривязывает даунстрим-задачи, у которых `dependsOn` указывал на теперь-заброшенный taskId (`grep` по `findByDependsOn*` — пусто, такого запроса нет). После Ф3-фикса (реальная проверка мержа вместо `done`) риск ещё выше: даунстрим будет тихо стоять в `queued` бесконечно, без алармов.

### Ф5 — `ClientDeliverableReadinessService` никогда не засчитывает recovery-цепочку в исходный client-deliverable — ПОДТВЕРЖДЕНО, фикс найден
`computeForProject` матчит `derivedTasks` по буквальному `sourceWishlistId` оригинального client-вишлиста. Recovery-таск живёт под **новым** `source_wishlist_id` — даже если он смержится, `mergedDeliverables` для оригинала не увеличится никогда. И P0.3-гейт, и falsification 90%-гейт могут остаться заблокированы навечно, даже когда работа фактически сделана через recovery.
**Фикс проверен на данных модели**: `AutoMergeService`'s recovery-вишлист явно наследует `featureId` (`recovery.setFeatureId(task.getFeatureId())`), а `FeatureService.resolveOrCreateFeatureId` **сохраняет** уже выставленный `featureId`, не создаёт новый (прочитал код построчно). Значит recovery-таск гарантированно окажется под тем же `featureId`, что и оригинальный client-deliverable — можно достоверно считать "смержено" по принадлежности к фиче, а не по буквальному wishlistId.

### Ф6 — новая, ранее не пройденная комбинация путей в AutoMergeService rebase-фиксе — ПОДТВЕРЖДЕНО И ИСПРАВЛЕНО
Прочитал `closeTaskAsFailed` и `claimSpecificTask` полностью. `closeTaskAsFailed` освобождает старый `ClaimEntity` (`releasedAt=now`). Код rebase-retry ставил `task.status=claimed` напрямую и звал `dispatch()` **без** `claimSpecificTask` — то есть ни один новый `ClaimEntity` никогда не создавался. `handlePrOpenedWorkflow`'ин implementer-ветка проверяет `claimService.hasActiveClaim(taskId)` (= есть ли `ClaimEntity` с `releasedAt IS NULL`) перед вызовом `claimService.complete(...)` — без активного claim это всегда `false`, значит даже полностью успешный ребейз произвёл бы реальный PR, который наш же пайплайн никогда не смог бы довести дальше `claimed`. Этот путь вообще ни разу не исполнялся до вчерашнего фикса (`dispatch()` всегда блокировался старой сессией) — баг был невидим, потому что код никогда не доходил до этой точки.
**Исправлено**: `task.status` теперь ставится в `queued` (не `claimed` напрямую — `claimSpecificTask`'ин лок-запрос требует именно `queued`), затем вызывается `claimService.claimSpecificTask(taskId, accountId)` (создаёт `ClaimEntity`, сам переводит в `claimed`), и уже обновлённый таск передаётся в `dispatch()` — та же последовательность, что везде в `dispatchQueuedTasks`.

### Ф7-Ф9 (без изменений с прошлого прохода, не переисследовались повторно в этом проходе)
- Паттерн A не обобщён на реальные implementer-таски (`closeLoopAndCreateFollowUps`).

## 2. Дизайн исправлений (проверенный, не "и так сойдёт")

### Д1 — редизайн `.eneik/`-резолюции (заменяет P0.1 ночной версии)
Убрать требование "ВСЕ файлы диффа под `.eneik/`". Вместо этого: взять пересечение `files ∩ {f.startsWith(".eneik/")}` — **подмножество**, не всё множество. Если непусто — попытаться синхронизировать каждый такой файл с main, независимо от того, что ещё есть в диффе (реальный код не трогается вообще). Дать этому **ровно одну** попытку на конфликт (`conflict.getResolutionAttempts() == 0`, до входа в обычный счётчик) — если после синка мерж всё равно не проходит (значит конфликт был не только/не в `.eneik/`), больше не зацикливаться на этой ветке, штатно уйти в существующий rebase/эскалацию путь. Без этого ограничения — риск бесконечного цикла "синк не помог → синк снова" на PR с реальным конфликтом где-то ещё.

### Д2 — общий "genuinely merged" хелпер вместо `TaskStatus.done`
Вынести `isTaskMerged(taskId)` из `ClientDeliverableReadinessService` в переиспользуемое место (например статический метод в `TaskRepository` или отдельный маленький сервис), использовать его и в `dispatchQueuedTasks`'ином `dependsOn`-чеке, и в readiness-сервисе. Один источник истины "смержено", а не два разных определения "готовности" в двух местах.

### Д3 — repoint `dependsOn` при эскалации/заброшенности — РЕДИЗАЙН при реализации
Исходная идея (перепривязать FK `dependsOn` на новый replacement-таск в момент эскалации) оказалась физически невозможной: в момент эскалации существует только recovery-`WishlistEntity`, реального replacement-taskId ещё нет (он появится позже, асинхронно, когда `tryCompileWishlistCheaply` его скомпилирует). Вместо перепривязки FK — сделан self-healing чек `ClientDeliverableReadinessService.isDependencySatisfied(TaskEntity dependency)`: зависимость считается удовлетворённой, если она сама смержена (`isTaskMerged`), ИЛИ если она `TaskStatus.failed` (заброшена) и существует ЛЮБОЙ другой таск с тем же `featureId` и той же ролью, который смержен. Recovery-wishlist гарантированно наследует и `featureId`, и роль оригинала (проверено в коде), так что это работает без какой-либо явной перепривязки, и естественно проходит через цепочку из нескольких эскалаций подряд (A заброшен → B тоже заброшен → C смержен — чек для A сразу найдёт C, не только B). Использован в `dispatchQueuedTasks` вместо прямого `isTaskMerged`.

### Д4 — readiness по `featureId`, не по буквальному `sourceWishlistId`
В `ClientDeliverableReadinessService.computeForProject`: для каждого client-deliverable вишлиста брать его `featureId` (через `resolveOrCreateFeatureId` или уже сохранённое значение), считать deliverable смерженным, если **любой** таск с этим `featureId` смержен — не только таск с буквально совпадающим `sourceWishlistId`. Использовать тот же Д2-хелпер.

### Д5 — доисследовать Ф6 перед тем как считать AutoMergeService rebase-фикс безопасным
Прочитать `closeTaskAsFailed` и `dispatch()` полностью, решить: нужен ли explicit `claimSpecificTask` перед `dispatch()` в `AutoMergeService`'ином rebase-retry коде (сейчас его нет, в отличие от `dispatchQueuedTasks`).
