**Пропускная способность:** 38 задач всего на момент остановки — 7 `done`, 1 `spike_completed` (терминально, не подлежит мерджу по природе — архитектурное решение), 20 `claimed` (в работе), 10 `queued`. PR: 5 открыто, 26 закрыто, **7 реально смёржено**. Ни одного ошибочного/мусорного мерджа за весь прогон не найдено.

**Дубли/мусор в названиях — важно разделять два разных явления:**
- **Одинаковые названия у РАЗНОГО реального содержимого** ("API Slice" ×11, "PR Review Fallback" ×11, "Compile Wishlist Into" ×7 — почти 3/4 всех задач): это косметика системы генерации коротких role-based названий (`TaskTitleBuilder`), не признак фабрикации — проверял содержимое конкретных "API Slice"-задач в T+85, все разные по JTBD. Встроенный анти-дубликат детектор (`DUPLICATE TASK TITLES`) ложно тревожился на это постоянно. Частично исправлено сегодня (уникальные суффиксы для 3 системных типов задач, которые я добавил), "API Slice"-паттерн — не трогал, отдельный вопрос.
- **Реальная содержательная дублирующая работа** — только один подтверждённый случай, но серьёзный: баг `RoleAdviceLoopService`, при недоступном Gemini эхом копировавший весь оригинальный 7288-символьный бриф в "рекомендацию", которая затем уходила на повторную декомпозицию через eneikdru. Именно это, а не название, объясняет взрывной рост 7→39 wishlist / 8→38 задач за 2 часа. **Исправлено и задеплоено** до конца эксперимента.

**Честная оценка Lean/TOC/Six Sigma по всему прогону:**
- Первая, настоящая декомпозиция клиентского ТЗ (4 задачи, T+40) — качественная: специфичная техническая лексика из брифа, разные Kano/Lean/Cynefin значения, измеримые 6σ-метрики. Не шаблон.
- Concern-driven задачи (PostgreSQL/Java21 фиксы) — тоже содержательные, прослеживаемая цепочка concern→wishlist→task→PR.
- Системный изъян, повторяющийся во ВСЕХ задачах без исключения за весь прогон: поле "TOC constraint ref" везде на деле — короткая метка подсистемы ("Dashboard state rendering", "Message variation processing"), а не найденное узкое место в терминах Theory of Constraints. Название поля вводит в заблуждение о том, что реально в нём хранится. Не исправлено — не патч, а вопрос к дизайну поля.
- Худшее по качеству — содержимое, порождённое багом RoleAdviceLoopService (fallback-шаблон "Recommendation based on task completion: [весь бриф]") — это ровно тот случай "заглушка вместо анализа", который просили ловить. Пойман, задокументирован, исправлен.

**Заметки для сравнения с внутренним мониторингом:**
- `GET /command-dashboard`'s `julesSessions` показывал пусто, когда сессии реально были активны (проверено на живых данных) — расхождение между `/dashboard` (правильно) и `/command-dashboard` (неверно), не исправлено.
- `DesignAssetService`/Stitch работает полностью без единой строки логов — единственный способ узнать, что что-то произошло, это смотреть на диск контейнера. Внутренний мониторинг через логи это в принципе не увидит.
- `AiHealthTracker`/`aiHealth` в `/api/system-status` — глобальный, не фильтруется по `?projectId=`, хотя параметр принимается — легко спутать чужие ошибки со своими.
- Falsification-история (`FalsificationRunEntity`) по-прежнему не имеет REST-эндпоинта — фиксируется в БД, но не видна нигде на дашборде.

**Что реально исправлено и задеплоено за время эксперимента (не после):**
1. `RoleAdviceLoopService` — не фабрикует fallback-рекомендацию при недоступном Gemini.
2. `spring.task.scheduling.pool.size=10` — вероятный фикс голодания редких задач планировщика.
3. Уникальные названия системных задач (компилятор/аудит/fallback-ревью) — устраняет ложные дубль-алярмы.
4. Уточнены логи: номер PR в вердикте fallback-ревью, честная строка при пропуске повторного диспетча.

**Сознательно отложено (полная версия, не патч, по прямому решению оператора):** путь Stitch-мокапов до реального использования (коммит в репозиторий, отдельная дизайн-ревью с философской/ролевой атакой, две папки черновик/утверждено) — первая задача для следующего чистого эксперимента.

**Итоговый вердикт как строгого аудитора:** система в целом работает — сквозной путь от клиентского ТЗ до смёрженного кода подтверждён многократно, новый Gemini-independent контур (eneikdru-компилятор + fallback-ревьюер) показал себя рабочим и содержательным, а не косметическим. Но эксперимент поймал два настоящих, воспроизводимых бага (не гипотетических), оба — прямое следствие недоступности Gemini, оба обнаружены только благодаря непрерывному внешнему наблюдению, а не встроенным метрикам системы. Внутренний мониторинг в текущем виде не заметил бы ни один из них: ни рост дублей (детектор ложно шумит на не-дубли и потому легко игнорируется), ни голодание планировщика (нет вообще никакой метрики "когда `@Scheduled`-задача последний раз реально выполнялась vs когда должна была").

Цикл мониторинга можно останавливать.
