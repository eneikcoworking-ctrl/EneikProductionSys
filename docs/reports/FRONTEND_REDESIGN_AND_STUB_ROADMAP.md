# Спецификация редизайна фронтенда и Дорожная карта устранения заглушек
**Система:** Eneik Production System
**Целевая аудитория:** Пользователь системы (упрощенный, чистый B2B-интерфейс без визуального шума)
**Статус документа:** СОГЛАСОВАНО (APPROVED)
**Автор:** Jules (Роль: BARCAN-TAG-11 · CLIENT-PERCEPTION)

---

## Введение

На основе обратной связи от пользователя текущий интерфейс перегружен информацией, дублирует списки проектов и содержит визуальный шум. Целью данной спецификации является:
1. **Переосмысление структуры фронтенда** в три чистые, специализированные страницы.
2. **Скрытие второстепенной информации** (завершенные, замороженные и архивные проекты скрываются или сворачиваются).
3. **Исключение любых заглушек/эмуляций** из метрик — отображаются только реальные, верифицированные данные.
4. **Диагностика всех заглушек (stubs/mocks)** в бэкенд-коде и составление детального плана их замены на реальные интеграции.

## ЧАСТЬ 1: Спецификация 3-страничного интерфейса фронтенда

### СТРАНИЦА 1: Главный пуль управления (Active Project Dashboard)
*Фокус страницы: один активный проект. Никакого визуального мусора.*

#### 1. Управление проектами (Скрытие шума):
* **Что убираем:** Из верхней и средней панели полностью удаляются списки неактивных проектов ( waiting, frozen, accepted, archived).
* **Как делаем:** Показывается **только 1 текущий активный проект** крупным заголовком. Все остальные проекты "сворачиваются" в аккуратный выпадающий список (аккордеон) в самом низу страницы под заголовком *"Архив и другие проекты (Свернуто)"*, который по умолчанию закрыт. Если активного проекта нет, выводится только форма создания нового проекта.

#### 2. Информационные блоки (Размещение):
Страница делится на 3 чистых горизонтальных или вертикальных блока:
1. **Блок проекта (Шапка):**
   * Название активного проекта, статус подключения к репозиторию.
   * Кнопки действий: *"Оркестрация"* и *"Принять проект"*.
2. **Блок "Пожелания клиента" (Wishlist):**
   * Форма ввода пожелания клиента (простой инпут + кнопка "Добавить").
   * Компактный список добавленных пожеланий с индикацией статуса (в работе / ожидает).
3. **Блок "Задачи проекта с приоритетом Кано" (Kano Tasks):**
   * Таблица или список задач, полученных в результате декомпозиции.
   * **Важно:** Каждая задача размечается по категориям модели **Кано** (*Must-Have* / Обязательные, *One-Dimensional* / Линейные, *Attractive* / Привлекательные, *Indifferent* / Безразличные).
   * Задачи сортируются по уровню приоритета (приоритет рассчитывается бэкендом на основе TOC-ограничений и модели Кано). У каждой задачи отображается ее текущий статус выполнения агентами Jules.

### СТРАНИЦА 2: Панель реальных метрик (Structured Metrics View)
*Фокус страницы: 100% реальные метрики, структурированные по ролям и процессам. Никаких заглушек.*

#### Структура группировки метрик:

```
[Реальные метрики системы]
   ├── Блок 1: Системный конвейер (Pipeline & Bottlenecks)
   ├── Блок 2: Качество и дефекты (Quality & DPMO)
   └── Блок 3: Ролевые метрики (12 BARCAN Roles)
```

#### Блок 1: Системный конвейер (Pipeline & Bottlenecks)
* **Задачи в очереди:** Реальное количество задач со статусом `queued`.
* **Задачи в работе:** Реальное количество задач со статусом `claimed` и `in_progress`.
* **Анализ узких мест (Bottlenecks):** Вывод активных ограничений TOC (каких ролей не хватает для разбора очереди задач, какие агенты простаивают). Время ожидания старейшей задачи в минутах (`oldestWaitingMinutes`).

#### Блок 2: Качество и дефекты (Quality & DPMO)
* **Показатель дефектов ворот качества (Gate DPMO):** Настоящее число ошибок прохождения Quality Gate на миллион возможностей.
* **Индекс конфликтов слияния (Merge Conflict DPMO):** Реальное количество сбоев автоматического слияния PR из-за конфликтов в гите к общему числу попыток слияния.
* **Активные конфликты:** Список реальных конфликтов с именами конфликтующих файлов и именами агентов, которые над ними работали.

#### Блок 3: Ролевые метрики (12 BARCAN Roles)
Метрики собираются динамически по результатам работы каждого специалиста:
1. **TAG-00 (Code Guardian):** Размер самого большого файла в кодовой базе (God-file detection, лимит 1000 строк).
2. **TAG-01 (Actualist Object):** Соблюдение MVC-структуры (соотношение классов контроллеров и репозиториев).
3. **TAG-03 (Designer) & TAG-11 (Frontend):** Метрики Web Vitals (LCP, CLS, FID) из отчетов Lighthouse/Axe, а также статус выполнения дизайн-проверок.
4. **TAG-05 (Necessary Identity):** Количество обнаруженных уязвимостей или незамаскированных секретов/токенов в коммитах.
5. **TAG-06 (Deontic Consistency):** Покрытие кода тестами (реальное количество Unit-тестов в JUnit, а не заглушки).
6. **TAG-07 (Second-Order Knowledge):** Количество ошибок, обработанных глобальным исключением, процент успешных API-ответов.
7. **TAG-09 (Moral Dilemma / Tech Lead):** Коэффициент декомпозиции пожеланий клиента в задачи (отношение wishes к tasks).

### СТРАНИЦА 3: Админка интеграций и токенов (Admin Settings)
*Фокус страницы: конфигурирование ключей, токенов и пула агентов.*

#### 1. Блок токенов интеграций:
* Поля ввода и маскированного отображения для:
  * **GitHub Token** (доступ к репозиториям).
  * **Linear API Key** (синхронизация задач).
  * **Jules API Key** (управление ключами AI-агентов).
* Кнопка *"Проверить соединение"* для каждой интеграции, отправляющая реальный ping-запрос к API сервиса и выводящая статус (Online / Offline) вместо статических плашек.

#### 2. Управление пулом аккаунтов (Account Pool):
* Форма добавления нового аккаунта агента Jules (имя, GitHub-логин, персональный API-ключ, роли).
* Таблица текущих запущенных агентов, их статусов (`idle`, `busy`, `offline`) и их реальной активности (Heartbeats).

## ЧАСТЬ 3: Дорожная карта реализации изменений (Roadmap)

```
[Шаг 1: Редизайн UI] ──> [Шаг 2: Удаление бэкенд-заглушек] ──> [Шаг 3: Тестирование и Запуск]
```

### Шаг 1: Изменение разметки и UX фронтенда (Интерфейс)
* **Спринт 1.1:** Переработка `App.svelte` и `CommandDashboardV2.svelte` для скрытия неактивных проектов. Реализация секции архива.
* **Спринт 1.2:** Создание страницы 2 (`MetricsView.svelte`). Перенос всех метрик, разбиение их на блоки по 12 BARCAN ролям и системным очередям. Удаление заглушек-чисел.
* **Спринт 1.3:** Создание страницы 3 (`AdminView.svelte`). Размещение там формы настройки токенов и списка пула аккаунтов.

### Шаг 2: Реализация бэкенд-интеграций (Удаление заглушек)
* **Спринт 2.1:** Замена "mock_diff" на реальный Git/GitHub API Diff Parser.
* **Спринт 2.2:** Подключение реальной проверки доступности Spring Boot Actuator Health вместо симуляций в `FalsificationCycleService`.
* **Спринт 2.3:** Настройка обязательной валидации токенов при загрузке системы (блокировка переходов, если ключи не заданы в админке).

### Шаг 3: Интеграционное тестирование
* Проверка взаимодействия фронтенда и бэкенда при выключенных и включенных токенах.
* Верификация адаптивности и разметки.
