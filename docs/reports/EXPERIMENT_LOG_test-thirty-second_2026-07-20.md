### 4. Duplicate "forced unblock" message sent to the same session twice within 2 seconds (found+fixed 01:15, operator caught it live in the Jules UI)

Operator spotted, directly in the Jules UI, two near-identical "Eneik orchestrator nudge" messages sent back-to-back to the UI Slice session, and separately flagged a deeper design concern: does the 30-minute staleness-based force-unblock risk interrupting sessions that are legitimately still working (just verbose, not stuck)?

**Root cause of the duplicate send**: `JulesDispatchService.runSessionSafetyMaintenance()` (which includes `forceUnblockOverflowedSessions()`) was being invoked from **two independent `@Scheduled` methods** - `JulesDispatchService.detectStuck()` (its own dedicated trigger, `jules.detect-stuck-rate-ms`, default 60s) AND `ContinuousOrchestrationService.continuousOrchestrate()` (`orchestration.rate-ms`, also default 60s). Spring's scheduled-task overlap prevention only guards a task against overlapping with *itself* - it does nothing to prevent two *different* scheduled methods from both calling the same shared logic concurrently. With both timers firing roughly every 60s independently (unsynchronized), they periodically landed close enough together that both transactions read the same session as "eligible" before either committed its `forcedUnblockAttempts`/`blindCycleCount` update, and both sent the nudge. Log evidence: sends at 01:07:15 and 01:07:17, 2 seconds apart, for session `sessions/5541652875502541487`.

**Fix**: removed the redundant `julesDispatchService.runSessionSafetyMaintenance()` call from `ContinuousOrchestrationService.continuousOrchestrate()` - the maintenance sweep already runs independently via `detectStuck()`'s own schedule, so nothing is lost, only the duplicate trigger is removed. Also fixed a related cosmetic bug found while reading this code: the async send's log line unconditionally said "Sent forced blind-overflow unblock message" even when the OTHER message variant (the "revising, review-feedback-pending" text) was the one actually sent - now labels correctly based on which branch fired.

**On the deeper design concern (some sessions legitimately need >30 min, can our nudges interrupt them)**: confirmed this is a REAL structural gap, not fully closed by the fixes above - `lastProgressAt` (the staleness clock) can only be refreshed via a genuine status transition or a newly-detected distinct question in `answerAgentQuestions`, but the latter path is entirely skipped while a session is in overflow (`activitiesOverflow=true`), so a session that goes blind and then keeps working hard and productively (exactly what happened to the UI Slice session - it was mid-implementation, not stuck) has no way to refresh its own staleness clock until it either finishes (reaches pr_opened) or shrinks back under the byte cap. The raised 10MB cap (fix #3) substantially reduces how often this triggers at all, but doesn't fully close the gap for genuinely long, verbose sessions that still exceed even the higher cap. **Not fixed yet - flagged as an open design question for the operator**: possible directions are (a) use a lighter-weight liveness signal that doesn't require the full activity payload (e.g., Jules's own lightweight session-status timestamp) to refresh `lastProgressAt` even while blind, or (b) raise `jules.stuck-threshold-minutes` specifically for the blind-overflow path, separate from the revising-session path which has a different, legitimate reason to want a tighter timeout. Deferred pending operator input rather than redesigning staleness detection unilaterally mid-experiment.

## 5. ROOT CAUSE of the whole night's redispatch storm, found and fixed (01:38-01:54)

Traced through `ClaimService.complete()` and found the actual structural bug behind everything in findings #1 and the operator's "13 merged PRs, mostly noise" alarm: **`ClaimService.complete()` is a two-call state machine** designed for the real implementer→reviewer lifecycle - the first call (task not yet `review`) moves a task to `TaskStatus.review`; only a SECOND call (task already `review`) advances it to `TaskStatus.done`. This is correct for real code tasks (implementer session completes -> review; a separate reviewer session later completes -> done). **System/carrier tasks have no second "reviewer" phase** - their one session IS the entire result - so every one-shot completion handler (`completeWishlistCompilation`, `completeFalsificationAudit`, `completeReviewerFallback`, `completeCoverageAudit`, `completeDesignReview`) called `claimService.complete()` exactly once, which left every single one of them **permanently parked at `TaskStatus.review`, never actually terminal**.

That `review` status was never really terminal: `ProjectFlowService.dispatchReviewTasks` scans EVERY task sitting at `TaskStatus.review` with no active review session and dispatches a **fresh reviewer session** for it - with zero awareness that a system task isn't real implementer code awaiting review. It just re-ran the SAME compiler/design-review/coverage-audit prompt from scratch, indefinitely. **This is the actual mechanism behind the entire evening's "duplicate design review" (finding #1), the pile of "Compile Wishlist"/"Coverage Audit Plan" PRs the operator counted (13 merged, mostly process), and the merge-conflict noise** - confirmed live: design-review task `21bfaa54` was fully re-completed 3 times over ~50 minutes (PR #6, #9, #14) before an unrelated retry-count/max-attempts path finally landed it on `failed`.

**Fix, two layers (operator: "реши эту проблему раз и навсегда" - solve this once and for all)**:
1. **At the source**: new `JulesDispatchService.markSystemTaskDone(task)` helper, called immediately after `claimService.complete()` in all 8 call sites across the 5 system-task completion handlers (4 in `completeWishlistCompilation` alone, covering its early-return/duplicate-discard/success/max-retry branches). Sets the task to the real terminal `TaskStatus.done` instead of leaving it at `review`.
2. **Defense in depth**: `dispatchReviewTasks` itself now explicitly skips any task matching `isWishlistCompilerTask`/`isFalsificationAuditTask`/`isReviewFallbackTask`/`isDesignReviewTask`/`isCoverageAuditTask`/`isPersistentWorkerCarrierTask`, so even a task that reaches `review` through some other path can never be redispatched this way again.

**Verified live post-deploy**: zero new "Dispatched task" log lines for the 4 known-failed system tasks in the following 8+ minutes; task count stable (no growth from redispatch); no new "Compile Wishlist"/"Coverage Audit"/"Design Review" PRs beyond legitimate new cycles. A later new Design Review task (`e220ccba`, for a genuinely new mockup) completed normally and correctly landed on `done` (not `review`) - the fix working as intended on a fresh case.

## 9. Fix applied 03:xx UTC - "principle of charity" for persistent worker sessions

Operator's framing: our force-unblock logic should trust Jules's rationality more - a session doesn't go quiet for no reason, it either already has an answer or has a question (Davidson's principle of charity). Directly matches the repeating bug from section 8: `lastProgressAt` only refreshed on a session-status transition, so a persistent worker that pushed a real, valid result file while Jules's external API kept reporting RUNNING (never transitioning to SUCCEEDED) looked "stuck" to us and got force-closed 29+ min after real work landed - confirmed to have happened twice in 15 minutes (review-fallback worker at 02:57, wishlist-compiler worker at 03:00).

**Fix 1 (JulesDispatchService.java):** before `forceUnblockOverflowedSessions` closes a persistent-worker carrier session as "exhausted without progress," it now calls a new read-only check `persistentWorkerHasReadyAnswer(session, task)` - looks up the session's actual open PR on GitHub and tries to parse the real result file (`.eneik/review-verdict.json` / `.eneik/task-plan.json`) directly from the PR branch, independent of what our local session.status says. If a valid, parseable batch is already sitting there, that's treated as decisive real progress: `lastProgressAt` is refreshed and `completePersistentWorkerCycle` is invoked immediately to process it (same logic the normal pr_opened edge trigger uses), instead of closing the session. Only proceeds to close if there is genuinely no ready answer.
