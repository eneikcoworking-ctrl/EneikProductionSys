## Метод

Каждая запись строится по одинаковым read-only источникам:
- `GET /api/projects/{id}/dashboard`, `/command-dashboard` — задачи, wishlist, статус
- `GET /api/projects/{id}/pull-requests` — реальное состояние PR через GitHub API
- `GET /api/projects/{id}/recent-activity?limit=60` — лог, скоуп-тегированный под проект
- `GET /api/system-status?projectId=...` — systemHealth, aiHealth, accounts
- `docker compose logs backend --since <окно>` — точечно, при необходимости деталей

В каждой записи фиксирую:
1. Что изменилось с прошлой записи (новые задачи/wishlist/PR/сессии)
2. Проверку названий задач/wishlist на дубли, бессмыслицу, нарушение границ ролей
3. Отдельно — качество Lean/TOC/Six Sigma полей: реальный анализ или шаблон
4. Любые ошибки/аномалии в логах
5. Свою оценку: движется ли работа, застряла ли, и почему
6. **Специально в этом эксперименте:** держать в фокусе — не создаётся ли работа до реального запроса (Lean-фикс #6), не переаудитятся ли одни и те же PR в falsification (Lean-фикс #7), реально ли доходят дизайн-мокапы до фронтенд-исполнителя через approved-папку.
