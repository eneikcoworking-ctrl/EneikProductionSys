# Диагностика производительности и причин долгой загрузки вкладки «Метрики»
**Система:** Eneik Production System - API `GET /api/system-status`
**Статус документа:** ДИАГНОСТИРОВАНО (VERIFIED)
**Автор анализа:** Jules (Роль: BARCAN-TAG-11 · CLIENT-PERCEPTION / Frontend & Performance Engineer)

---

## Введение и Суть проблемы

При переходе на вкладку **«Метрики»** (Страница 2) пользователь сталкивается с экстремально долгой задержкой загрузки (время ожидания ответа от сервера может составлять от нескольких секунд до десятков секунд в зависимости от объема накопленных исторических данных в базе).

С точки зрения **пользовательского опыта (UX)**:
* Задержка более **1.0 секунды** прерывает непрерывную ментальную связь пользователя с интерфейсом.
* Задержка более **10 секунд** воспринимается мозгом как «зависание» системы, заставляя пользователя нервничать, повторно кликать по вкладке (что еще сильнее перегружает бэкенд повторными запросами) или закрывать страницу (синдром *Spinner Fatigue*).

Статический анализ кода класса `SystemStatusService.java` позволил выявить **три фундаментальные архитектурные проблемы бэкенда**, которые в сумме приводят к деградации производительности.

---

## Подробный технический аудит узких мест (Bottleneck Analysis)

### 1. Проблема №1: Классический N+1 SQL-запрос (N+1 Query Problem)
* **Локализация:** Метод `linearCompleteness()` в `SystemStatusService.java`.
* **Как работает код сейчас:**
  ```java
  List<TaskEntity> tasksWithLinear = taskRepository.findAll().stream()
          .filter(task -> task.getLinearIssueId() != null && !task.getLinearIssueId().isBlank())
          .toList();
  ...
  for (TaskEntity task : tasksWithLinear) {
      LinearIssueMetadataEntity metadata = linearIssueMetadataRepository.findById(task.getId()).orElse(null);
      ...
  }
  ```
* **Почему это тормозит:**
  Бэкенд сначала делает один большой запрос, чтобы выгрузить все задачи. Затем во внутреннем цикле для **каждой задачи**, у которой есть привязка к Linear, делает **отдельный индивидуальный запрос** в таблицу `linear_issue_metadata` по первичному ключу.
  * Если в системе зарегистрировано **500 задач**, бэкенд выполнит **1 + 500 = 501 последовательный SQL-запрос** к БД!
  * Это создает колоссальные накладные расходы на сетевые транзакции, компиляцию запросов СУБД и блокировки соединений.

---

### 2. Проблема №2: Выгрузка всех сущностей в память (JPA `findAll` Memory Bloat)
* **Локализация:** Методы `qualityGate()`, `linearCompleteness()` и `conflictDpmo()`.
* **Как работает код сейчас:**
  Для подсчета суммарного количества успешных/проваленных проверок ворот качества бэкенд запрашивает:
  ```java
  for (TaskEntity task : taskRepository.findAll()) {
      JsonNode report = task.getQualityGateReport();
      ...
  }
  ```
  Аналогично в `conflictDpmo()` запрашиваются абсолютно все PR-ревью и все зарегистрированные конфликты:
  ```java
  List<PrReviewEntity> allReviews = prReviewRepository.findAll();
  List<TaskConflictEntity> allConflicts = taskConflictRepository.findAll();
  ```
* **Почему это тормозит:**
  Метод `findAll()` заставляет Hibernate/JPA поднять из базы данных **все до единой строки** таблиц `tasks`, `pr_reviews` и `task_conflicts`, включая огромные JSON-объекты (такие как `qualityGateReport` и полные payloads задач), и распарсить их в Java-объекты.
  * Это приводит к моментальному исчерпанию лимита оперативной памяти JVM (Heap Memory) при росте базы.
  * Вызывает жесткие просадки CPU из-за работы сборщика мусора (Garbage Collector), который пытается очистить тысячи тяжелых объектов, созданных на доли секунды для обычного суммирования.
  * Если в архиве лежит 5000 завершенных задач, бэкенд будет судорожно выгружать все 5000 задач каждый раз, когда пользователь заходит на вкладку метрик.

---

### 3. Проблема №3: Последовательные циклические агрегатные запросы
* **Локализация:** Метод `tasks()` в `SystemStatusService.java`.
* **Как работает код сейчас:**
  ```java
  for (TaskStatus status : TaskStatus.values()) {
      counts.put(status, taskRepository.countByStatus(status));
  }
  ```
* **Почему это тормозит:**
  Вместо одного быстрого группирующего запроса, бэкенд поочередно делает **6 отдельных запросов `SELECT COUNT(*)`** к базе данных (по одному на каждый статус). Это создает 6 последовательных блокирующих сетевых обращений к СУБД.

## Дорожная карта оптимизации и устранения тормозов (Action Plan)

Чтобы вкладка метрик открывалась мгновенно (менее чем за **50-100 миллисекунд**), необходимо переписать расчеты метрик в `SystemStatusService.java` с Hibernate-объектов на чистые агрегаты СУБД:

### Шаг 1: Устранение N+1 в `linearCompleteness` через JOIN (Срочно)
Вместо цикла с `findById` выгружать статистику одним запросом с левым соединением (LEFT JOIN) таблиц `tasks` и `linear_issue_metadata`.
```sql
SELECT t.id, m.pr_url, m.blockers, m.dod_text
FROM tasks t
LEFT JOIN linear_issue_metadata m ON t.id = m.task_id
WHERE t.linear_issue_id IS NOT NULL AND t.linear_issue_id != ''
```
Это вернет всю нужную информацию за **1 запрос** вместо 500.

### Шаг 2: Расчет DPMO в СУБД без загрузки JSON в память Java
Вместо выгрузки тысяч тяжелых сущностей `TaskEntity` для парсинга JSON, выполнять нативный SQL запрос, извлекающий только заполненные отчеты:
```sql
SELECT quality_gate_report FROM tasks WHERE quality_gate_report IS NOT NULL
```
Или, если СУБД поддерживает JSON-функции (H2/PostgreSQL), выполнять весь расчет DPMO прямо внутри базы данных за миллисекунды.

### Шаг 3: Переход на агрегатный GROUP BY для статусов задач
Заменить 6 отдельных `countByStatus` запросов на один группирующий SQL-запрос:
```sql
SELECT status, COUNT(*) FROM tasks GROUP BY status
```
Это позволит бэкенду мгновенно получить распределение задач по всем статусам за один проход.

### Шаг 4: Кеширование тяжелых метрик (Metrics Caching)
Поскольку исторические метрики (такие как DPMO за все время или за последние 7 дней) меняются не каждую секунду, их расчет следует обернуть в аннотацию `@Cacheable(value = "systemMetrics", unless = "#result == null")` со временем жизни кеша (TTL) около **1-5 минут** или обновлять по планировщику (`@Scheduled`) в фоновом режиме.

## Заключение

Медленная загрузка метрик — это чисто технический дефект бэкенд-реализации, не связанный со сложностью интерфейса Svelte. Фронтенд работает максимально эффективно и отрисовывает полученные данные мгновенно. Перевод расчетов `SystemStatusService.java` на эффективные SQL-запросы и кеширование полностью уберет задержку, вернув плавность и комфорт при использовании системы Eneik.
