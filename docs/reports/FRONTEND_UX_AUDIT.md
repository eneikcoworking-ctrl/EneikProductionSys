## 1. Введение и Исполнительное резюме (Executive Summary)

Целью данного аудита является всесторонний анализ текущего состояния фронтенда (Svelte) системы **Eneik Production System** исключительно с точки зрения пользовательского опыта (UX), эргономики, когнитивной психологии восприятия и стандартов доступности (WCAG 2.1). Код фронтенда был детально исследован без внесения изменений во избежание дестабилизации текущего релиза.

### Ключевой вывод (Epistemic Status: VERIFIED)
Текущий фронтенд обладает сильной архитектурной основой и современным технологическим стеком (Svelte 5, Vite, TypeScript), однако содержит **ряд критических визуальных регрессий, серьезных нарушений адаптивности и критических барьеров доступности (WCAG)**, которые существенно снижают качество B2B SaaS продукта и могут приводить к когнитивной усталости и ошибкам пользователей.

## 2. Анализ критических дефектов интерфейса

### 2.1. Критическая визуальная регрессия: Фантомный Tailwind CSS
* **Статус обнаружения:** ВЕРИФИЦИРОВАНО (VERIFIED) — Прямой анализ файлов `package.json`, `index.html`, `ClientDeliveryView.svelte`, `ProjectActiveCard.svelte`, `ProjectCreateForm.svelte` и `ProjectHistoryList.svelte`.
* **Суть проблемы:**
  Компоненты доставки клиенту и управления проектами написаны с активным использованием утилит-классов Tailwind CSS (например, `space-y-8`, `max-w-4xl`, `mx-auto`, `p-6`, `flex`, `justify-between`, `text-3xl`, `bg-green-600`, `rounded-full` и др.). Однако в проекте **полностью отсутствует сборщик Tailwind CSS** (нет зависимостей в `package.json`, нет конфигурации `tailwind.config.js`), а также **не подключена CDN-версия** в `index.html`.
* **Влияние на UX и когнитивное восприятие:**
  В реальном окружении эти компоненты отображаются абсолютно **без стилей**, используя дефолтное оформление браузера (unstyled). Списки накладываются друг на друга, шрифты имеют гигантский или микроскопический размер, кнопки выглядят как стандартные серые прямоугольники из 90-х годов. Это полностью разрушает доверие пользователя к качеству продукта (нарушение *Закона Якоба* о привычных качественных интерфейсах) и делает страницу "Client Delivery View" визуально непригодной для демонстрации клиентам.
* **Рекомендация:**
  1. Либо полностью интегрировать Tailwind CSS в проект (`npm install -D tailwindcss postcss autoprefixer` и настройка `vite.config.ts`),
  2. Либо переписать стили данных компонентов на стандартный CSS, соответствующий переменным из `frontend/src/app.css` (Design System), чтобы сохранить чистоту DOM и исключить лишние зависимости.

### 2.2. Нарушение адаптивности: Схлопывание сетки Dashboard на мобильных экранах (`responsive_ok` Fail)
* **Статус обнаружения:** ВЕРИФИЦИРОВАНО (VERIFIED) — Анализ стилей компонента `CommandDashboardV2.svelte`.
* **Суть проблемы:**
  Главная рабочая панель использует сетку `.dashboard-grid`:
  ```css
  .dashboard-grid {
    display: grid;
    grid-template-columns: repeat(3, 1fr);
    gap: var(--space-6);
    flex: 1;
    min-height: 0;
  }
  ```
  При этом в файле **отсутствуют медиа-запросы (`@media`)**, которые перестраивали бы эту сетку на мобильных экранах (375px - 768px).
* **Влияние на UX и когнитивное восприятие:**
  При просмотре панели с мобильного устройства или планшета, три тяжелые информационные колонки ("Client Wishlist", "Quality Gates & Delivery", "Team Pool & Queue") сжимаются в три вертикальные полосы шириной около 100-110 пикселей каждая. Текст переносится по одной букве, интерактивные кнопки накладываются друг на друга, а списки задач превращаются в нечитаемую кашу. Это делает невозможным использование системы с мобильного телефона, нарушая базовый принцип доступности и пространственной интуиции Кристофера Пикока.
* **Рекомендация:**
  Добавить медиа-запрос в `CommandDashboardV2.svelte` для экранов шириной менее `1024px`, перестраивающий сетку в одну колонку:
  ```css
  @media (max-width: 1024px) {
    .dashboard-grid {
      grid-template-columns: 1fr;
    }
  }
  ```

### 2.3. Нарушение доступности: Скрытие контура фокуса (WCAG SC 2.4.7 Focus Visible)
* **Статус обнаружения:** ВЕРИФИЦИРОВАНО (VERIFIED) — Исследование файла `app.css`.
* **Суть проблемы:**
  В глобальном файле стилей для элементов ввода и выпадающих списков жестко прописано скрытие стандартного фокус-контура браузера:
  ```css
  .filter-group select { outline: none; }
  .textarea-wrapper textarea { outline: none; }
  ```
  При этом альтернативные стили фокуса (например, `border-color` или `box-shadow`) для клавиатурной навигации через клавишу `Tab` не настроены, либо настроены только для мыши.
* **Влияние на UX и доступность:**
  Пользователи, перемещающиеся по интерфейсу с клавиатуры (включая незрячих и слабовидящих людей, использующих скринридеры, а также профессиональных операторов, предпочитающих быстрый клавиатурный ввод), не видят, на каком интерактивном элементе в данный момент находится фокус. Это делает интерфейс неиспользуемым и дискриминационным (прямое нарушение принципа инклюзивного интерфейса Марты Нуссбаум).
* **Рекомендация:**
  Удалить `outline: none;` или заменить его на стильный, высококонтрастный фокус-контур, использующий переменные нашей дизайн-системы:
  ```css
  .filter-group select:focus-visible,
  .textarea-wrapper textarea:focus-visible,
  button:focus-visible {
    outline: 2px solid var(--accent);
    outline-offset: 2px;
  }
  ```

### 2.4. Нарушение доступности: Отсутствие семантической связи форм (WCAG SC 1.3.1 Info and Relationships)
* **Статус обнаружения:** ВЕРИФИЦИРОВАНО (VERIFIED) — Исследование `AdminDashboard.svelte`.
* **Суть проблемы:**
  Форма добавления нового аккаунта Jules использует текстовые параграфы `<p class="label">` в качестве подписей к полям ввода:
  ```html
  <p class="label">New Account Name</p>
  <input bind:value={newName} placeholder="e.g. Jules-08" />
  ```
  Здесь нет тега `<label>` и отсутствует связь по `id` между подписью и полем ввода.
* **Влияние на UX:**
  Экранные дикторы (Screen Readers) не могут ассоциировать текстовую подпись с полем ввода. Когда пользователь переходит в поле ввода, скринридер озвучивает его просто как "пустое поле ввода, редактирование", не сообщая, что именно туда нужно вводить.
* **Рекомендация:**
  Использовать валидную семантическую разметку:
  ```html
  <label for="new-account-name" class="label">New Account Name</label>
  <input id="new-account-name" bind:value={newName} placeholder="e.g. Jules-08" />
  ```

## 3. Анализ когнитивной эргономики

### 3.1. Закон Миллера: Информационная перегрузка Панели Администратора
* **Теоретический базис:** Закон Миллера гласит, что средний человек может удерживать в рабочей памяти лишь $7 \pm 2$ элементов одновременно.
* **Проблема во фронтенде:**
  Компонент `AdminDashboard.svelte` представляет собой единый непрерывный холст, на котором одновременно отображаются:
  1. Общие метрики системы (5 крупных показателей).
  2. Интеграции (форма с кнопками изменения/сохранения токенов).
  3. Форма создания аккаунта Jules (4 поля ввода + кнопка).
  4. Таблица аккаунтов (многоколоночный список с кнопками деактивации и редактирования ключей).
  5. Списки коллабораторов GitHub (4 колонки).
  6. Метрики DPMO и конфликты слияния (карточки трендов + списки активных конфликтов).
  7. Итоговая панель статуса.
* **UX Эффект:**
  Администратор испытывает высокую когнитивную нагрузку. Найти нужную настройку или быстро оценить состояние пула аккаунтов становится сложно из-за визуального "шума".
* **Рекомендация:**
  Разбить панель администратора на вкладки (Tabs) или использовать сворачиваемые секции (`<details>` / аккордеоны) для группировки информации по смысловым блокам:
  - **Вкладка 1:** Общий статус и Интеграции (GitHub, Linear).
  - **Вкладка 2:** Управление пулом Jules-аккаунтов (Создание, список, статусы).
  - **Вкладка 3:** Аналитика конфликтов (DPMO, Активные конфликты слияния).

### 3.2. Закон Фиттса: Размер интерактивных элементов на мобильных устройствах
* **Теоретический базис:** Время достижения цели зависит от расстояния до нее и размера самой цели. Интерактивные элементы должны быть достаточно большими для быстрого и безошибочного нажатия.
* **Проблема во фронтенде:**
  - В `AdminDashboard.svelte` кнопки редактирования API-ключей `.mini-btn` имеют крайне малый размер (шрифт `10px`, паддинг `2px 6px`). Высота кнопки составляет менее `24px`.
  - Инпуты и выпадающие списки в некоторых блоках не имеют минимальной высоты и сжимаются при уменьшении экрана.
* **UX Эффект:**
  Пользователь на мобильном устройстве (или даже на десктопе с тач-падом) будет промахиваться мимо кнопок, случайно нажимая соседние строки или ссылки. Это ведет к раздражению и ошибочным действиям (например, случайный клик по "Decommission" вместо "Edit").
* **Рекомендация:**
  Увеличить все интерактивные кнопки до минимального размера кликабельной зоны `44px` по высоте и ширине (стандарт Apple iOS / Google Material Design). Для десктопных "микро-кнопок" увеличить внутренний отступ (padding) и добавить `outline-offset` для визуальной изоляции.

---

### 3.3. Гештальт-принципы: Нарушение Сходства и Близости
* **Проблема во фронтенде:**
  - **Статусы проектов:** В `App.svelte` используются плашки статусов со специфическими стилями:
    ```css
    .project-strip button .badge.active { background: #d1fae5; color: #065f46; }
    .project-strip button .badge.accepted { background: #dbeafe; ... }
    ```
    При этом в `AdminDashboard.svelte` для статусов используются совершенно другие классы и цвета (`.idle`, `.busy`, `.offline`).
  - **Языковой диссонанс:** В `App.svelte` статусы выводятся на английском языке (`active`, `waiting`, `accepted`, `analyzing`), в то время как в `ProjectActiveCard.svelte` они переводятся на русский (`Активен`, `Завершен`). Это нарушает принцип предсказуемости и консистентности.
* **UX Эффект:**
  Пользователь тратит дополнительное когнитивное время на декодирование одного и того же логического статуса в разных частях системы.
* **Рекомендация:**
  Создать единый справочник статусов на уровне фронтенда и использовать унифицированный компонент `<Badge status={status} />`, гарантирующий одинаковый цвет, форму и язык отображения во всех панелях.

## 4. Конкретный план оптимизации и дорожная карта (Roadmap)

Для достижения высшего уровня качества интерфейса (FPPI = 100%) рекомендуется выполнить следующие шаги:

```
[Этап 1: Исправление Сборки] ──> [Этап 2: Адаптивность и Сетки] ──> [Этап 3: Доступность и Скринридеры]
```

### Этап 1: Интеграция стилей (Срочно)
1. Настроить Tailwind CSS в `frontend/package.json` и `vite.config.ts`, чтобы все Tailwind-классы в `ClientDeliveryView.svelte` и других компонентах начали компилироваться корректно.
2. Альтернативно: Заменить все классы вроде `text-3xl`, `bg-green-600` на семантические CSS переменные дизайн-системы, прописанные в `app.css`.

### Этап 2: Оптимизация мобильного отображения
1. Добавить адаптивные брейкпоинты для всех Grid-контейнеров на мобильных экранах (375px).
2. Ограничить ширину таблиц в панели администратора и добавить горизонтальный скролл только для внутренних данных, а не для всей страницы.

### Этап 3: Полноценное внедрение WCAG 2.1
1. Вернуть видимый фокус-контур для клавиатурной навигации (`:focus-visible`).
2. Обеспечить корректные теги `<label>` для всех элементов форм.
3. Проверить коэффициент контрастности текста (минимум 4.5:1 для обычного текста по стандарту WCAG AA).

## Заключение
Проведенный аудит показывает, что визуальная эстетика системы Eneik движется в правильном направлении, но технические недоработки реализации (отсутствие Tailwind при его использовании в коде, жесткие сетки без медиа-запросов и скрытие фокуса) критически сдерживают потенциал продукта. Внедрение рекомендаций данного отчета позволит повысить индекс перцептивного качества фронтенда (FPPI) до целевых 100%.
