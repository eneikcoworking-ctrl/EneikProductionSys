# Словарь терминов (жёстко зафиксировано, 2026-07-21)

Возник разнобой в терминах ещё в начале ночной сессии 2026-07-20/21: "фича" в бытовом понимании оператора — это одна сессия/одна ветка/один PR, а в коде `FeatureEntity`/`featureId` означает совсем другое — группу из нескольких задач. Договорённость: не переименовывать код (`FeatureEntity` остаётся как есть, слишком много мест это ломает), а зафиксировать словарь для разговора и документации так, чтобы термины не путались.

## Вишлист (Wishlist / `WishlistEntity`)

Сырой, ещё не разбитый на работу запрос. **Вход** в компилятор, не сама работа.

- `source`: `client` (от заказчика) / `role` / `role_mismatch_followup` (recovery после провала/конфликта) / `chaotic_debt` / `self_falsification` / `onboarding_finding` / `idle_generation` / `coverage_gap`.
- `status`: `pending` → `compiling` → `converted_to_task` (или `dismissed`).
- Компилятор (persistent worker или one-shot Jules-сессия) читает вишлист и раскладывает его на одну или несколько Задач.

## Задача (Task / `TaskEntity`)

Атомарная единица исполняемой работы. **В норме: одна Jules-сессия = одна ветка = один PR.** Это то, что оператор в быту называет "фича" — одна сессия, одна ветка.

- Привязана к одной роли (`role.tag`, например `BARCAN-TAG-08`).
- Может зависеть от другой Задачи через `dependsOn` (одна задача, не список).
- Статусы: `queued → claimed → review → done` (реализация), либо `pending_review → review → done` (после ревью), либо `failed`/`blocked` при провале.
- **`TaskStatus.done` ≠ "смержено"** — `done` ставится при апруве ревью, факт мержа проверяется отдельно (`ClientDeliverableReadinessService.isTaskMerged`). Это различие — источник как минимум двух багов, найденных и исправленных 2026-07-21 (Ф3, Ф5).

## Эпик (в коде — Feature / `FeatureEntity` / поле `featureId`)

**То, что оператор просил называть "эпик", чтобы не путать со своим бытовым "фича".** Группа из нескольких связанных Задач, произошедших от одного корневого Вишлиста через декомпозицию (`ProjectFlowService.buildTaskGraphFromSlices`).

Пример из сегодняшней ночи: вишлист "Project Notes" разложился на 4 задачи (Data Schema, API Contract, Backend Endpoints, UI Slice) — все четыре под **одним** `featureId`. Recovery-задачи (созданные после эскалации провалившегося конфликта) **тоже наследуют этот же `featureId`** — эпик переживает провал/замену отдельных задач внутри себя. Именно на этом свойстве построены сегодняшние фиксы Ф4 (`isDependencySatisfied`) и Ф5 (readiness по featureId вместо буквального `sourceWishlistId`).

- `FeatureEntity.rootWishlistId` — исходный вишлист, из которого эпик родился.
- Нет отдельного статуса "эпик завершён" как поля — завершённость эпика сейчас определяется агрегированно, через `ClientDeliverableReadinessService.computeForProject` (все ли задачи эпика реально смержены).

## Как это связано

```
Вишлист (client brief)
   │  компиляция (buildTaskGraphFromSlices)
   ▼
Эпик (featureId) ──┬── Задача 1 (Data Schema)     ── 1 сессия, 1 ветка, 1 PR
                    ├── Задача 2 (API Contract)    ── 1 сессия, 1 ветка, 1 PR
                    ├── Задача 3 (Backend Endpoints)── 1 сессия, 1 ветка, 1 PR
                    └── Задача 4 (UI Slice)         ── 1 сессия, 1 ветка, 1 PR
```

Если Задача 1 проваливается и эскалируется — рождается **новая** Задача (recovery, `role_mismatch_followup`) под **тем же** Эпиком, но с новым `id` и новым `source_wishlist_id`. Старый термин "фича" в бытовом смысле оператора соответствует именно **Задаче**, не Эпику.
