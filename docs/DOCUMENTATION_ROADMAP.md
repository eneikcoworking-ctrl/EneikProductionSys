# ДОРОЖНАЯ КАРТА РЕОРГАНИЗАЦИИ ДОКУМЕНТАЦИИ (DOCUMENTATION ROADMAP)

Данный документ представляет собой стратегический план по наведению порядка в документации проекта **EneikProductionSys**. В ходе аудита было выявлено большое количество разрозненных, частично устаревших (legacy) и дублирующих друг друга файлов, описывающих гипотетическое состояние системы, в то время как проект уже перешел в стадию полнофункционального MVP.

---

## 1. Текущие проблемы и болевые точки

1. **Разрозненность файлов:** В папке `docs/` находится более 20 markdown-файлов без четкой структуры каталогов.
2. **Устаревшие сведения (Legacy vs. Fact):** Ряд документов (например, `docs/SYSTEM_DIAGNOSIS.md`, `docs/AGENCY_PROTOCOL.md`) описывают систему как пустой шаблон с "Hello World", в то время как в коде уже реализована сложная распределенная оркестрация агентов, гейты качества и микросервис машинного обучения.
3. **Терминологические расхождения:** Роль `BARCAN-TAG-09` в одних файлах описывается как "Technical Product Manager", в других — как "Technical Lead" (конфликтуя с `BARCAN-TAG-00 Code Guardian`).
4. **Отсутствие "Единого источника правды":** Новым разработчикам или агентам сложно понять, как именно взаимодействуют backend (Spring Boot), frontend (Svelte) и ML-сервис (FastAPI) в рамках сквозного бизнес-процесса.

## 2. Целевая структура папки `docs/`

Предлагается распределить все документы по пяти функциональным категориям (директориям):

```text
docs/
├── architecture/          # Архитектурные спецификации и технический дизайн
│   ├── MVC_ARCHITECTURE_STANDARD.md
│   ├── MVC_LOGICAL_DESIGN.md
│   ├── SCHEMA_CONTRACT.md
│   ├── SEQUENCE_FLOW.md
│   └── openapi.yaml
│
├── business/              # Бизнес-функции, требования и бэклог продуктов
│   ├── AGENT_TASKS_BACKLOG.md
│   ├── PRODUCT_FACTORY_BACKLOG.md
│   └── GAP_ANALYSIS.md
│
├── agents/                # Протоколы взаимодействия ИИ-агентов
│   ├── AGENCY_PROTOCOL.md          # (Обновленный протокол оркестрации)
│   ├── ROLE_EXCELLENCE_CHARTER.md   # (Хартия ролей)
│   └── AI_REVIEW_GUIDELINES.md     # (Инструкции для ИИ-ревьюера)
│
├── guides/                # Инструкции по запуску, эксплуатации и онбордингу
│   ├── OPERATING_GUIDE.md
│   └── SYSTEM_OVERVIEW.md          # (Новый мастер-документ "Единый источник правды")
│
└── reports/               # Результаты аудитов, тестов и анализов (Логи изменений)
    ├── ROLE_USE_AUDIT_REPORT.md    # (Фактический аудит использования ролей)
    ├── MVC_AUDIT_REPORT.md
    ├── FRONTEND_UX_AUDIT.md
    ├── METRICS_PERFORMANCE_DIAGNOSIS.md
    └── E2E_VERIFICATION_REPORT.md
```

## 3. План консолидации и архивации

Для очистки корневой папки `docs/` и устранения дублирования предлагается выполнить следующие действия:

| Исходный файл | Статус | Действие |
|---|---|---|
| `docs/SYSTEM_DIAGNOSIS.md` | **Устарел** | Консолидировать в новый `docs/SYSTEM_OVERVIEW.md` и удалить как дубликат. |
| `docs/SIMPLE_STEPS_RISKS.md` | **Устарел** | Перенести полезные выводы в `docs/architecture/MVC_ARCHITECTURE_STANDARD.md`, файл удалить. |
| `docs/JULES_AUDIT_MANIFEST.md` | **Дубликат** | Интегрировать в `docs/reports/ROLE_USE_AUDIT_REPORT.md` (раздел о Fail-Open уязвимостях) и `docs/agents/AI_REVIEW_GUIDELINES.md`, файл удалить. |
| `docs/TASK_SPECIFICATION_SCHEMA.md`| **Актуален**| Перенести в `docs/architecture/SCHEMA_CONTRACT.md`. |
| `docs/CONTINUOUS_IMPROVEMENT.md`| **Актуален**| Перенести в `docs/guides/OPERATING_GUIDE.md` как раздел "Непрерывное улучшение процессов". |

## 4. Создание Мастер-документа `docs/SYSTEM_OVERVIEW.md`

Главным шагом в наведении порядка является создание **Master System & Business Function Reference** (`docs/SYSTEM_OVERVIEW.md`). Этот документ должен содержать:
1. **Общее бизнес-описание:** Какую бизнес-задачу решает система (автоматическая декомпозиция требований пользователя, claim-оркестрация задач между ролевыми агентами, контроль качества на гейтах, автоматический авто-мерж).
2. **Архитектурный ландшафт:** Схема взаимодействия 3 сервисов (Spring Boot @ 8080, Svelte @ 3000, Python FastAPI ML @ 8000).
3. **Жизненный цикл задачи (Task Lifecycle):** Подробное описание переходов статусов (`queued` -> `claimed` -> `review` -> `done`/`failed`) и работы Lease Watchdog (освобождение зависших задач).
4. **Анализ уязвимостей контура контроля:** Описание проблемы Fail-Open (когда падение ML-сервиса приводит к автоматическому одобрению PR с заглушкой `"Antigravity fallback review pass."`) и перехода к Fail-Safe архитектуре.

## 5. Ожидаемый результат

В результате выполнения этого плана проект получит:
* Четкую, интуитивно понятную структуру документации.
* Устранение ложных описаний ("Hello World" вместо реальной оркестрации).
* Единую точку входа для разработчиков и бизнес-пользователей.
* Документальное закрепление реальных механизмов контроля качества кодовой базы.
