# ДОРОЖНАЯ КАРТА РЕОРГАНИЗАЦИИ ДОКУМЕНТАЦИИ (DOCUMENTATION ROADMAP)

Данный документ представляет собой стратегический план по наведению порядка в документации проекта **EneikProductionSys**. Мы полностью удалили устаревшие (legacy) файлы, привели роли к единому пониманию и четко разделили файлы на две категории: рантайм-активные (используемые бэкендом и кодом) и чисто справочные (описывающие систему).

---

## 1. Единое понимание ролей (Unified Role Understanding)

В ходе реорганизации была устранена терминологическая путаница между ролями:
* **BARCAN-TAG-00 (CODE-GUARDIAN):** Исключительно **Tech Lead / Code Review**. Отвечает за финальное слияние, контроль размера файлов (блокировка God-files >50КБ), цикломатическую сложность и качество кода.
* **BARCAN-TAG-09 (MORAL-DILEMMA):** Исключительно **Technical Product Manager / Mediator**. Является прагматическим медиатором и единственным владельцем бизнес-конвейера, обладающим правом компиляции требований из вишлиста (`createTaskFromWishlist`). Обеспечивает баланс скорости и качества согласно TOC (Теории Ограничений) и Lean.

Все 12 ролей теперь имеют строго фиксированные детерминированные границы и зоны ответственности, исключающие пересечение или дублирование функций.

---

## 2. Разделение файлов: Рантайм-активные vs. Справочные

Для предотвращения случайного нарушения логики работы бэкенда и ML-сервиса, все файлы разделены на две строгие группы.

### Группа 1: Рантайм-активные файлы (Используются кодовой базой)
*Эти файлы напрямую загружаются бэкендом (Java/Spring Boot) или аналитическим контуром (Python/FastAPI) во время работы приложения. Их перемещение или удаление строго ЗАПРЕЩЕНО без изменения соответствующего кода.*

1. **Файлы манифестов ролей (`/BARCAN-TAG-XX_*.md`):**
   - **Расположение:** Корень репозитория.
   - **Использование:** Загружаются службой `RoleCapabilityLoader` по путям, прописанным в БД (таблица `roles`, колонка `rules_path`, задаваемая в миграции `V3__core_agency_schema.sql`). Парсятся классом `RoleRulesParser` для извлечения критериев отказа (`REFUSAL CRITERIA`). Также сканируются Python-скриптом в ML-сервисе.
2. **Руководство ИИ-ревьюера (`docs/AI_REVIEW_GUIDELINES.md`):**
   - **Расположение:** `docs/AI_REVIEW_GUIDELINES.md`.
   - **Использование:** Загружается бэкендом в `JulesDispatchService.java` для передачи инструкций внешнему ИИ-сервису при проведении автоматического код-ревью PR.
3. **Стандарт дизайн-системы (`docs/DESIGN_SYSTEM.md`):**
   - **Расположение:** `docs/DESIGN_SYSTEM.md`.
   - **Использование:** Проверяется в `TechnicalLeadCompiler.java` (метод проверяет наличие ссылки на этот файл в DoD для всех UI-задач).

---

### Группа 2: Чисто справочные файлы (Описывают систему и процессы)
*Эти файлы не читаются кодом бэкенда или ML-сервиса. Их можно свободно реорганизовывать, редактировать, перемещать в соответствующие папки или удалять по мере устаревания.*

1. **Мастер-обзор системы (`docs/SYSTEM_OVERVIEW.md`):** Главный источник правды, описывающий архитектуру, claim-жизненный цикл и уязвимости контура контроля.
2. **Карта директорий (`docs/DIRECTORY_MAP.md`):** Логическое закрепление MVC-границ за различными аккаунтами.
3. **Диаграмма последовательности (`docs/SEQUENCE_FLOW.md`):** Описание сетевых и транзакционных вызовов.
4. **Контракт схем (`docs/SCHEMA_CONTRACT.md`):** Описание структуры БД и интеграционных интерфейсов.

## 4. Дальнейшие шаги по реорганизации каталогов

В соответствии с планом, неактивные справочные файлы из Группы 2 будут постепенно перемещены в специализированные подпапки каталога `docs/` (например, `docs/architecture/`, `docs/business/`, `docs/guides/`) при плановых обновлениях структуры документации, гарантируя, что ни один исполняемый путь Группы 1 не будет затронут.
