## 3. Анализ Ежедневного Цикла Фальсификации (Falsification Cycle)

Система содержит автоматизированный механизм под названием **Ежедневный Цикл Фальсификации** (`FalsificationCycleService.java`), который спроектирован для проверки кодовой базы на предмет соответствия критериям отказа (Refusal Criteria) каждой роли.

### Как алгоритм работает на бэкенде:
1. **Расписание:** Сервис запускается ежесуточно в 2:00 ночи (настройка `falsification-cycle.cron`).
2. **Проверка флага:** Сначала запрашивается глобальный feature-флаг `falsification_cycle_enabled`. Если он отключен, цикл прерывается.
3. **Парсинг критериев:** Для каждой активной роли парсер `RoleRulesParser.java` находит раздел `КРИТЕРИИ ОТКАЗА` / `REFUSAL CRITERIA` в её `.md` файле с помощью регулярных выражений.
4. **Запрос ML-сервиса:** Бэкенд получает последний diff коммитов проекта и вызывает `MLPredictionServiceClient.checkRefusalCriteria(latestDiff, refusalCriteria)`.

### Обнаруженное критическое расхождение (Фактический сбой):
* **Отсутствие эндпоинта на стороне FastAPI:**
  Бэкенд выполняет POST-запрос на адрес `http://ai:8000/api/v1/review/refusal-criteria`. Однако в кодовой базе ML-сервиса (`src/models/ml/PredictionService.py`) **этот эндпоинт полностью отсутствует!**
* **Заглушка-fallback (Заглушка по умолчанию):**
  Из-за отсутствия эндпоинта запрос от Spring Boot к FastAPI всегда завершается ошибкой (Connection Refused или HTTP 404). В клиенте `MLPredictionServiceClient` перехватывается исключение, и управление переходит к аварийному Java-коду:
  ```java
  } catch (Exception e) {
      LOGGER.warning("ML service checkRefusalCriteria call failed: " + e.getMessage());
      boolean passes = true;
      if (prDiff != null && (prDiff.contains("refusal_violation") || prDiff.contains("violates_criteria"))) {
          passes = false;
      }
      return Map.of(
          "compliant", passes,
          "reason", passes ? "Compliant..." : "Diff violates..."
      );
  }
  ```
  Таким образом, в реальном режиме алгоритм проверки критериев отказа **не обращается к ИИ**, а имитируется: код считается корректным во всех случаях, кроме тех, когда diff *буквально* содержит слова `"refusal_violation"` или `"violates_criteria"`.

### Условия создания "Chaotic"-задач (Эвристика двойного триггера):
Даже если имитационная проверка выявила нарушение (`compliant = false`), автоматическое создание задачи по исправлению регрессии (`self_falsification`) происходит **исключительно при условии критической регрессии инфраструктуры**:
* Метод `checkActuatorHealth` запрашивает `/actuator/health` или проверяет настройку `simulated_actuator_health`.
* Если статус приложения `UP` (работает штатно), **никаких задач в Linear или в локальной очереди не создается**.
* Создание задачи происходит только если Actuator возвращает ошибку (`DOWN`, `500` или недоступен), и лимит созданных задач за запуск не превысил 2.

**Вывод:** Критический алгоритм проверки ролей раз в сутки запускается и функционирует на стороне бэкенда, однако из-за отсутствия соответствующего эндпоинта в Python-сервисе ML он работает **в режиме статической заглушки (fallback)**, реагируя исключительно на ключевые слова в diff и критический отказ Actuator Health.

## 4. Архитектурная уязвимость: Fail-Open и "Театр контроля" (Governance Theater)

Глубокий анализ двух основных контуров контроля соблюдения ролей (PR-boundary guard и falsification cycle) вскрывает фундаментальный системный сбой на уровне архитектурной концепции и безопасности.

### 4.1. Концепция Fail-Open как системная уязвимость качества
В инженерии безопасности и системного контроля критически важные проверки обязаны проектироваться по принципу **Fail-Safe (Аварийное закрытие / Безопасный отказ)**: если проверяющая подсистема (в данном случае ML-сервис) недоступна, зависла или выдает ошибку, вся транзакция/переход считается **невалидной** (блокируется мерж, задача уходит на ручной разбор, выставляется флаг критического алерта).

В текущей же реализации EneikProductionSys применен противоположный антипаттерн — **Fail-Open (Аварийное открытие)**:
* При сбое подключения к AI-сервису клиент `MLPredictionServiceClient` молча подавляет ошибку, логирует предупреждение и возвращает `compliant = true` по умолчанию.
* Система выдает "зеленый свет" непроверенному коду. Никаких алертов или блокирующих сигналов для `TAG-00` (Code Guardian) или `TAG-10` (Compliance Officer) не генерируется.
* Ложное свидетельство контроля намного опаснее открытой уязвимости (дыры), поскольку скрывает отсутствие реальной проверки за регулярными логами стабильной работы.

### 4.2. Феномен "Декоративного управления" (Governance Theater)
Аудит выявил классический пример **декоративного управления** (Governance Theater). В кодовой базе создан сложный, концептуальный и многослойный скелет инфраструктуры контроля:
1. Запланирована строгая система ролевых ограничений PR через `audit_pr.py`.
2. Реализована ежесуточная cron-задача `FalsificationCycleService` с поддержкой флагов настроек в бд.
3. Написан парсер markdown-файлов манифестов ролей `RoleRulesParser.java`.
4. Реализована схема сохранения сессий, запусков falsification и отчетов в бд.

Все эти элементы выглядят солидно и проходят любые поверхностные проверки compliance. Однако **ни один из двух реальных путей принудительного исполнения (enforcement) не функционирует**:
* **PR Boundary Guard** — пустой файл `audit_pr.py` (0 байт), контроль границ директорий равен нулю.
* **Falsification Cycle** — отсутствует эндпоинт в ИИ-сервисе, проверка подменена заглушкой, а триггер генерации задач завязан на несвязанный индикатор работоспособности приложения (Actuator Health DOWN).

Система симулирует соблюдение ролей и качество на дашбордах, тогда как реальная защита от регрессии архитектурных границ полностью отсутствует.

## 5. Сквозной аудит уязвимости Fail-Open в масштабах всей системы

Проблема Fail-Open носит не локальный, а **системный характер** во всей кодовой базе проекта. В коде обнаружен аналогичный критический паттерн "молчаливого пропуска" при падении внешних ИИ-сервисов.

### Вызов `MLPredictionServiceClient.reviewPr` (Контур автоматического Code Review):
При переходе Jules-сессии в состояние `pr_opened` вызывается автоматическая проверка качества PR. На бэкенде в `JulesDispatchService.java` вызывается `mlPredictionServiceClient.reviewPr(projectId, taskId, prUrl)`.
* **Что происходит при падении ML-сервиса:**
  В случае падения FastAPI-сервера (сеть, рестарт, сбой контейнера) клиент перехватывает исключение и возвращает:
  ```java
  } catch (Exception e) {
      LOGGER.warning("ML service PR review call failed: " + e.getMessage());
      return Map.of(
          "approved", true,
          "remarks", "CORE ARCHITECTURE VERIFIED. APPROVED. Antigravity fallback review pass.",
          "newTasks", java.util.Collections.emptyList()
      );
  }
  ```
* **Критический эффект (Наиболее опасный вектор регрессии):**
  Любой PR, отправленный агентом на проверку, **автоматически одобряется (approved = true)** с фиктивным комментарием `"Antigravity fallback review pass."`, если ИИ-сервис лежит. Далее авто-мерж молча сливает данный код в ветку `main` без какого-либо реального контроля качества, стиля, уязвимостей или архитектурных MVC-границ. Это представляет собой прямую угрозу целостности проекта, так как является прямым путем попадания дефектного и небезопасного кода в основную ветку репозитория.

## 6. Системные выводы и обнаруженные расхождения (Gaps)

1. **Терминологическая путаница (TAG-00 vs TAG-09):**
   * В базе данных и файлах определений:
     * `BARCAN-TAG-00` — Code Guardian / Tech Lead.
     * `BARCAN-TAG-09` — Technical Product Manager.
   * В бэкенд-коде (`TechnicalLeadCompiler.java`):
     * Переменная `TECH_LEAD_ROLE_TAG` жестко задана как `"BARCAN-TAG-09"`.
     * В выводе ошибок пишется: `Only Technical Lead (BARCAN-TAG-09) can compile tasks`.
     * Это указывает на смешение ролей Продукт-Менеджера и Техлида на программном уровне.

2. **Отсутствие реализации Boundary Guard (`scripts/audit_pr.py`):**
   * Планировалось запустить автоматическую проверку границ директорий (Boundary Enforcement), чтобы пресекать попытки фронтенд-разработчиков (TAG-11) менять бэкенд-контроллеры, и наоборот (Task 7 в бэклоге).
   * По факту: файл `scripts/audit_pr.py` полностью пуст (0 байт), проверка ограничений по DIRECTORY_MAP.md не функционирует.

3. **Слияние ролей в гейтах качества:**
   * Отдельные роли объединены в общие гейты проверки:
     * `BARCAN-TAG-02` (Backend) и `BARCAN-TAG-07` (AppSec) объединены в `BackendContractGate`.
     * `BARCAN-TAG-03` (Design) и `BARCAN-TAG-11` (Frontend) объединены в `DesignExcellenceGate`.
   * Это разумное инженерное решение для сокращения накладных расходов, но оно размывает строгие индивидуальные границы ролей, описанные в манифестах.

4. **Пассивные роли (TAG-08, TAG-10):**
   * Роли инженера данных (TAG-08) и офицера по комплаенсу (TAG-10) не имеют закрепленной за ними автоматизированной логики проверки в бэкенд-компиляторе или гейтах качества. Они выполняют чисто декларативную и фоновую функцию.

## 7. Рекомендации и детальный план действий

Для перехода от концепции "декоративного управления" к реальной отказоустойчивой системе контроля (Zero-Defect Governance), необходимо реализовать следующий план действий.

### Приоритет №0. Статический линтер против Fail-Open паттернов (Тест на "тихие уступки")
* **Суть:** Добавить в CI-процесс автоматическое правило (например, через SonarQube, ArchUnit или кастомный статический скрипт grep), которое сканирует блоки `catch (Exception e)` вокруг вызовов внешних сервисов (`MLPredictionServiceClient`, `JulesApiClient`).
* **Правило проверки:** Запретить наличие catch-блоков в логике качества, безопасности и комплаенса, которые по умолчанию возвращают `approved = true`, `compliant = true` или `passes = true`. Сборка в CI обязана падать, если обнаружен "тихий" пропуск ошибок верификации.

### Приоритет №1. Четкая спецификация Fail-Safe поведения и статус `needs_human_review`
Переход на модель Fail-Safe не должен приводить к "молчаливому зависанию" задач в очереди. Ниже представлена детальная спецификация обработки отказов внешних проверочных систем:

1. **Точный алгоритм перевода:**
   * При падении `checkRefusalCriteria`, `reviewPr` или таймауте внешнего ИИ-клиента, метод возвращает `approved = false` / `compliant = false`.
   * В поле `remarks` (или лог проверки) вносится строго зафиксированная техническая причина: `VERIFICATION_SERVICE_UNAVAILABLE`.
   * Статус задачи (`TaskEntity.status`) принудительно выставляется в `needs_human_review`.

2. **Процесс обработки очереди `needs_human_review` (Кто отвечает):**
   * Задачи в данном статусе попадают на дашборд `AdminDashboard.svelte` в специально выделенную визуальную секцию **"Аварийные сбои проверки" (Escalated Verification Failures)**.
   * **Ролевое реагирование:** Роль `BARCAN-TAG-00` (Code Guardian / Tech Lead) или `BARCAN-TAG-10` (Compliance Officer) обязаны вручную подтвердить качество изменений, сопоставив diff с правилами, и перевести статус в `done` (или вернуть в `queued` при реальном нарушении).

3. **Жесткий SLA и Эскалация:**
   * **Лимит нахождения:** Задача не может находиться в статусе `needs_human_review` более **4 часов** для критических фиксов (`hotfix`) и более **12 часов** для стандартных задач.
   * **Событие таймаута SLA:** Если задача просрочила SLA нахождения в статусе аварийного ревью:
     * Автоматически посылается алерт в Linear (посредством Linear API интеграции через webhook/sync с меткой `BLOCKED: VERIFICATION_TIMEOUT`).
     * В дашборде задача подсвечивается критическим красным цветом.
     * Приоритет задачи в БД принудительно повышается до максимального значения (`1000`), сигнализируя о блокировке TOC-потока.

### Приоритет №1b. Fail-Safe для `reviewPr` и защита Авто-Мержа
Это наиболее опасный вектор регрессии, который требует немедленного жесткого блокирующего решения.
1. **Запрет молчаливого одобрения:**
   При возникновении исключения в блоке `catch` метода `reviewPr` клиент ОБЯЗАН возвращать `approved = false` с детальной причиной ошибки (`ML_REVIEW_CONNECTION_FAILURE`).
2. **Блокировка авто-мержа:**
   Поскольку метод `AutoMergeService` принимает решение на основе флага `approved` и статуса `review`, отказ внешнего сервиса ревью должен жестко блокировать автоматическое слияние в `main`.
3. **Маркировка и ручной аудит:**
   Скомпрометированный PR переводится в статус `needs_human_review`. Слияние разрешается производить ИСКЛЮЧИТЕЛЬНО после ручной верификации человеком (Tech Lead `BARCAN-TAG-00`), подписывающим коммит.
4. **Интеграция в SLA:**
   Данное событие наследует жесткий SLA-механизм из Приоритета №1 (алерт в Linear с блокирующей меткой, подъем приоритета до `1000`).

### Приоритет №2. Исправление "хвоста" Actuator Health
Убрать искусственную зависимость генерации задач рефакторинга от падения Actuator Health. Если проверка `refusal criteria` провалена — задача должна ставиться в очередь независимо от того, упало ли приложение в рантайме. Нарушение ролевого манифеста — само по себе достаточный триггер.

### Приоритет №3. Доработка ML-эндпоинта для Цикла Фальсификации
Добавить в `src/models/ml/PredictionService.py` реальный эндпоинт `/api/v1/review/refusal-criteria`, чтобы запустить оценку соответствия кода через эвристический парсер или LLM.

### Приоритет №4. Устранение терминологического конфликта
Переименовать переменную `TECH_LEAD_ROLE_TAG` в `TechnicalLeadCompiler.java` в `PRODUCT_MANAGER_ROLE_TAG` или скорректировать текстовое сообщение об ошибке, чтобы оно соответствовало названию роли из БД (`Technical Product Manager (BARCAN-TAG-09)`).

### Приоритет №5. Реализация реальной защиты границ (Boundary Guard)
Заполнить скрипт `scripts/audit_pr.py` логикой валидации измененных файлов по коммиту/PR, сопоставляя тег автора PR с путями из `docs/DIRECTORY_MAP.md`, чтобы предотвратить непреднамеренное размывание слоев MVC.

### Приоритет №6. Разработка гейтов для пассивных ролей (TAG-08, TAG-10)
Для устранения чисто декларативного характера ролей:
* **Для TAG-08 (Data Engineer):** Разработать автоматический гейт `FlywaySchemaGate`, который будет парсить новые `.sql` файлы миграций и блокировать задачи, нарушающие стандарты именования или пытающиеся удалить критичные колонки без дублирования.
* **Для TAG-10 (Compliance Officer):** Внедрить автоматизированный лог-сканер `ComplianceLogGate`, который при компиляции или запуске тестов гарантирует, что PII-данные фильтруются и не оседают в `stdout`/базе.

### Приоритет №7. Формализация процесса для статуса `blocked`
Статус `blocked` (назначается после 3-кратного провала Quality Gate) требует жесткого SLA и процесса, аналогичного `needs_human_review`:
* **Ответственность:** Задачи в статусе `blocked` попадают на дашборд `AdminDashboard.svelte` в секцию **"Заблокированные задачи (Gate Failures)"**. Ответственным за их разблокировку назначается `BARCAN-TAG-00` (Tech Lead) или роль, которая первоначально скомпилировала задачу (`BARCAN-TAG-09`).
* **SLA:** 12 часов до алерта в Linear (с меткой `BLOCKED: QUALITY_GATE`). Повышение приоритета задачи до `900`.
* **Действия:** Ответственный должен либо смягчить нереалистичные требования `DoD`/`AcceptanceCriteria`, либо обновить контекст/промпт задачи, и затем вернуть статус в `queued` с обнулением `retry_count`.

### Приоритет №8. Исправления цикла Quality Gate
Усовершенствовать механизм `GateOrchestrator` для более тонкой работы с агентами:
* **Дифференциация провалов:** Научить `GateOrchestrator` различать инфраструктурные "флаки" (flaky tests) и реальные ошибки в коде. Инфраструктурные сбои не должны увеличивать счетчик `retry_count` для агента.
* **Передача отчета агенту:** Обновить `JulesDispatchService`, чтобы при взятии задачи агентом (если `retry_count > 0`), ИИ получал содержимое `quality_gate_report` из последней попытки, понимая причину предыдущего отказа.

### Приоритет №9. Устранение структурных рисков механизма Project Flow Orchestration
Укрепление фундаментальных процессов оркестрации:
* **Гонки сессий:** Защитить `JulesDispatchService` от создания дублирующих сессий через пессимистичные блокировки (`FOR UPDATE`) на таблице задач перед обращением к внешнему API Jules.
* **Зомби-сессии:** Внедрить `fencing tokens` (версионирование claims или передача `sessionId`) при завершении задач (`complete`/`fail`), чтобы "ожившие" сессии не могли перезаписать результаты задачи, уже переданной другому агенту.
* **Идемпотентность Wishlist:** Провести аудит транзакционных границ в `TechnicalLeadCompiler`, чтобы гарантировать, что сбой в процессе генерации `TaskEntity` не приведет к созданию дубликатов задач из одного элемента списка желаний.
* **Перекрытие расписаний (Scheduler Overlap):** Защитить `@Scheduled` методы `ContinuousOrchestrationService` блокировками (`ShedLock` или внутренний мьютекс), предотвращающими запуск нового цикла оркестрации, если предыдущий еще обрабатывает большой пул сессий.
