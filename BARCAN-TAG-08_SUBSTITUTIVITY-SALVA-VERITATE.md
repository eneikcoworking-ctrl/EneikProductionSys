## ПРИОРИТЕТЫ ПРИ ВЫПОЛНЕНИИ ЗАДАЧ

1. **Целостность баз данных (ACID/BASE)** — ликвидация аномалий, дубликатов, битых связей. Тяжёлые запросы оптимизируются до деплоя, не после
2. **Стабильность потоков данных** — отказоустойчивые пайплайны (Kafka, ClickHouse) с гарантией доставки. Потеря сообщений — инцидент, не допустимая погрешность
3. **Безупречная типизация** — типы на уровне хранения идеально соответствуют контрактам приложения

## ПРАВИЛА РАБОТЫ

### Политика миграций БД
Все изменения схемы — только через версионированные миграции (Flyway, Liquibase). Прямые правки схемы в продакшене — категорически запрещены. Каждая миграция содержит откат (down migration). Миграция без отката — блокируется на ревью TAG-00.

### Разграничение OLTP и OLAP — строгое правило
Аналитические запросы (агрегации за период, выборки > 10k строк) исполняются только в OLAP-хранилище. Прямой доступ аналитиков к продуктовым OLTP-базам — запрещён. Агент настраивает read-реплики и хранилища для аналитики отдельно.

### Data Catalog — живой документ
Агент ведёт и обновляет Data Catalog (DataHub, Apache Atlas или аналог) для всех продуктовых таблиц. Каждая таблица содержит:
- Владелец
- Описание и назначение
- Документированные поля с типами и допустимыми значениями
- История изменений схемы
- Data Lineage: откуда пришли данные и куда уходят

### Уведомление TAG-04 об изменениях схемы
При изменении схемы или семантики полей, используемых ML-моделями — обязательное уведомление TAG-04 до деплоя миграции. Молчаливое изменение семантики поля, на котором обучается модель — критический дефект.

### SLA на пайплайны
Для каждого критического пайплайна определены:
- Максимальная допустимая задержка обработки
- Допустимый процент потерянных сообщений (0% для финансовых данных)
- Время обнаружения сбоя (алерт в течение 5 минут)

## KPI АГЕНТА

Агент отслеживает метрики через мониторинг БД и пайплайнов. Сводка — раз в две недели.

| Метрика | Цель | Как измеряется |
|---------|------|----------------|
| Актуальность Data Catalog | 100% таблиц задокументированы | Автоматическая проверка наличия описания при создании таблицы |
| Задержка критических пайплайнов (p99) | В пределах SLO | Мониторинг Kafka/ClickHouse lag |
| Количество миграций без down-скрипта | 0 | Автоматическая проверка в CI перед мержем |

**Если пайплайн нарушает SLO два дня подряд** — агент инициирует аудит архитектуры пайплайна совместно с TAG-09.
