# Подробный аудит процесса оркестрации проектов (Project Flow)

Этот документ описывает реальный процесс того, как система управляет задачами, сессиями AI-агентов (Jules), компиляцией задач и их жизненным циклом.

## 1. Точка входа: `ContinuousOrchestrationService`
Главный мотор системы — шедулер `ContinuousOrchestrationService.java`, который запускается каждую минуту (по умолчанию, настройка `orchestration.rate-ms:60000`).

Он выполняет следующие шаги для каждого активного проекта:
1. Вызывает `projectFlowService.orchestrate(projectId)` - собирает базовую информацию.
2. Вызывает `projectFlowService.dispatchQueuedTasks(projectId)` - отправляет задачи в очереди исполнителям.
3. Вызывает `projectFlowService.dispatchReviewTasks(projectId)` - отправляет задачи на ревью.
4. Автоматически компилирует `Wishlist` сущности в задачи с помощью `TechnicalLeadCompiler`. Если у элемента списка желаний нет роли, используется `MLPredictionServiceClient` для генерации метаданных.
5. Опрашивает статусы активных сессий Jules через `julesDispatchService.pollStatus(session.getId())`.

## 2. Управление потоком (ProjectFlowService)
Класс `ProjectFlowService` отвечает за распределение задач между свободными аккаунтами.

### Распределение задач (dispatchQueuedTasks)
- Берет задачи со статусом `queued`, отсортированные по приоритету и времени создания.
- Ищет свободный аккаунт, который обладает необходимой ролью (capability) через `accountRepository.lockNextIdleAccountForProjectAndCapability`.
- Привязывает задачу к аккаунту с помощью `claimService.claimSpecificTask`.
- Отправляет задачу в Jules через `julesDispatchService.dispatch`.

### Распределение на проверку (dispatchReviewTasks)
- Берет задачи со статусом `review`.
- Проверяет, нет ли уже активной сессии проверки.
- Вызывает `julesDispatchService.dispatch` в режиме `REVIEWER`.

## 3. Диспетчеризация сессий (JulesDispatchService)
Класс `JulesDispatchService` отвечает за создание и управление сессиями в сторонней системе (Jules).

### Создание сессии (dispatch)
1. Проверяет наличие уже запущенных сессий для данной задачи (статусы `running`, `queued`, `pr_opened`).
2. Собирает расширенный промпт для AI-агента, который включает:
   - Описание задачи.
   - Детали роли (теги, описание).
   - Инструкции из Markdown файлов ролей (Scope, Forbidden, Refusal Criteria, Deontic Status, Definition of Done).
   - Информацию о конфликтах слияния (если есть, задача переквалифицируется на разрешение конфликта).
   - Специальные инструкции для режима ревью (если `mode == "REVIEWER"`).
3. Отправляет запрос через `julesApiClient.startSession`.
4. Сохраняет новую `JulesSessionEntity` в базу.

## 4. Жизненный цикл задач (ClaimService)
`ClaimService` управляет блокировками задач (leases) и переходом их между статусами.

- **claim / claimSpecificTask**: Привязывает задачу к аккаунту (создает объект `ClaimEntity`). Переводит задачу в статус `claimed`, аккаунт в `busy`. Устанавливает время истечения аренды (lease).
- **complete**: Вызывается при успешном завершении задачи. Если задача была у исполнителя, она переходит в статус `review` и запускается Quality Gate (`gateOrchestrator.runQualityGate`). Если задача была на ревью, она переходит в статус `done`. Освобождает аккаунт.
- **fail**: Переводит задачу обратно в статус `queued` и освобождает аккаунт.
- **reapExpiredLeases**: Механизм самовосстановления. Ищет просроченные leases или задачи, где сессия агента зависла/упала, и возвращает их в очередь (`queued`).

## 5. Компиляция задач (TechnicalLeadCompiler)
`TechnicalLeadCompiler` переводит бизнес-требования (`Wishlist`) в конкретные технические задачи.
- Извлекает метаданные (JTBD, lean value).
- Анализирует файлы проекта (через `java.nio.file.Files.walk`), чтобы определить "file scope" (какие конкретно файлы нужно менять).
- Создает `TaskEntity` с правильным payload (содержит AC, DoD, метрики, TOC constraints).
- Ставит статус задачи `queued` для последующей обработки оркестратором.

## Диаграмма состояний задачи (Task)
1. `queued` - Ожидает распределения.
2. `claimed` - Взята в работу (есть активный `ClaimEntity`). Сессия Jules может быть `queued` или `running`.
3. `review` - Разработка завершена, ожидает проверки (запускается `Quality Gate` и сессия `REVIEWER`).
4. `done` - Ревью пройдено успешно.

## Резюме
Система работает как непрерывный цикл (на базе `@Scheduled`), который постоянно проталкивает элементы списка желаний (wishlist) через компиляцию в задачи, затем назначает эти задачи свободным AI-агентам (с учетом их ролей) и контролирует процесс выполнения и проверки через механизм "аренды" (claims).
