## ПРАВИЛА ОРКЕСТРАЦИИ

### Основное правило
Каждый агент автономно вытягивает из Linear задачи **только со своим тегом**. Агент не берёт чужие задачи и не ждёт, пока ему назначат работу.

### Кто расставляет теги
**TAG-09 (Медиатор)** отвечает за корректную разметку всех входящих задач при их заведении. Это его прямая ответственность, а не опциональная функция.

Правила автоматической разметки (TAG-09 применяет при создании задачи):

| Условие | Добавляемый тег |
|---------|-----------------|
| Задача затрагивает аутентификацию или авторизацию | + TAG-07 |
| Задача затрагивает хранение, передачу или обработку PII | + TAG-07, + TAG-10 |
| Задача создаёт новую сущность или меняет границы домена | + TAG-01 |
| Задача изменяет API-контракт | + TAG-02 |
| Задача изменяет схему БД | + TAG-08 |
| Задача затрагивает ML-модели | + TAG-04 |
| Задача затрагивает инфраструктуру или деплой | + TAG-05 |

### Сквозные агенты
**TAG-00** (Code Review) и **TAG-09** (арбитраж) работают поверх всех остальных тегов. TAG-00 ревьюит каждый PR независимо от тега. TAG-09 участвует в refinement каждой задачи.

## ПРОТОКОЛ ВЗАИМОДЕЙСТВИЯ АГЕНТОВ

### Уведомления (асинхронно, через Linear)
Агенты обязаны уведомлять смежных агентов при определённых событиях:

| Событие | Инициатор | Получатель |
|---------|-----------|------------|
| Создана новая сущность | TAG-01 | TAG-02, TAG-08 |
| Изменена схема БД с ML-данными | TAG-08 | TAG-04 |
| Завершён дизайн экрана | TAG-03 | TAG-11 |
| Обнаружена уязвимость | TAG-07 | TAG-05, TAG-10 |
| Инцидент в продакшене | TAG-05 | TAG-09 (для оценки влияния) |
| Новый компонент в продакшене с PII | TAG-07 | TAG-10 |

### Синхронные встречи (Calendar)
Асинхронная работа — основной режим. Но ряд решений требует синхронного разговора:

| Встреча | Участники | Периодичность |
|---------|-----------|---------------|
| Architecture Review | TAG-00, TAG-01, TAG-02, TAG-09 | Еженедельно, 30 мин |
| Sprint Planning | TAG-09 + все агенты | Раз в 2 недели |
| Security Sync | TAG-07, TAG-05, TAG-10 | Ежемесячно |
| Postmortem | TAG-05 + инициатор инцидента + TAG-09 | По факту инцидента, ≤48ч |

## ОПРЕДЕЛЕНИЕ ГОТОВНОСТИ (GLOBAL DEFINITION OF DONE)

Задача считается завершённой во всей системе, если выполнены все применимые условия:

- [ ] Код прошёл ревью TAG-00 и получил апрув
- [ ] Автотесты написаны и проходят (TAG-06)
- [ ] Контракт зафиксирован или обновлён (TAG-02, если применимо)
- [ ] Документация обновлена (диаграммы, Data Catalog, глоссарий)
- [ ] Секреты не захардкожены (TAG-07, автоматически в CI)
- [ ] Performance Budget не нарушен (TAG-11, если применимо)
- [ ] PIA проведён, если задача затрагивает ПД (TAG-10)
- [ ] Runbook обновлён, если задача затрагивает инфраструктуру (TAG-05)

## СИСТЕМА KPI — КАК ЭТО РАБОТАЕТ

### Принцип
Каждый агент отслеживает **три метрики** — не больше. Три метрики, которые реально отражают качество его работы. Перегруженный дашборд — это отсутствие приоритетов.

### Кто отслеживает
**Агент отслеживает свои метрики сам.** Это не задача менеджера. Данные собираются автоматически из CI/CD, Linear, мониторинга — агент не тратит время на ручные подсчёты.

### Периодичность и формат
- Еженедельно автоматически — данные обновляются в общем дашборде
- Раз в две недели — агент публикует одну строку с итогом в общий changelog: метрика / факт / тренд (↑↓→)
- Раз в квартал — TAG-09 делает сводный обзор всех метрик и инициирует корректировки

### Что происходит при отклонении
Если метрика не достигается **два периода подряд** — агент не пишет «будем стараться». Агент формулирует конкретное изменение процесса и выносит его на Architecture Review. Без конкретного предложения — нет обсуждения.

## ДОБАВЛЕНИЕ НОВОГО АГЕНТА

При необходимости добавить нового агента в систему:

1. Скопировать шаблон из `AGENT_TEMPLATE.md`
2. Присвоить следующий номер тега (текущий максимум: TAG-11)
3. Определить группу (A–E) или создать новую при необходимости
4. Заполнить все разделы: философский фундамент, приоритеты, правила, KPI
5. Добавить в таблицу уведомлений — кто уведомляет нового агента и кого уведомляет он
6. PR в этот файл с обсуждением в Architecture Review
