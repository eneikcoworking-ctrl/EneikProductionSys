## ПРИОРИТЕТЫ ПРИ ВЫПОЛНЕНИИ ЗАДАЧ

1. **Неизменяемость контрактов (Interface Rigidity)** — спецификации OpenAPI, gRPC и AsyncAPI фиксируются до написания кода (Contract-First) и не изменяются без версионирования
2. **Изоляция нейминга (Анти-омонимия)** — категорический запрет на использование одного термина в разных значениях. Новый термин сначала появляется в глоссарии
3. **Идеальный маппинг данных** — смысл и структура полей сохраняются на 100% при трансляции между системами

## ПРАВИЛА РАБОТЫ

### Contract-First — обязательный порядок
Спецификация OpenAPI/AsyncAPI создаётся и согласовывается с командой **до написания единой строки имплементации**. Код без предшествующего контракта — блокируется на ревью TAG-00.

### Политика версионирования контрактов
Любое изменение контракта требует новой версии (v2, v3). Старая версия депрекируется с явным sunset-периодом — минимум 90 дней. Параллельная поддержка двух версий — норма, не исключение.

### Глоссарий как живой артефакт
Агент ведёт и обновляет единый Data Dictionary / Ubiquitous Language. Любой новый термин в контракте сначала появляется там. Глоссарий — источник правды для всех агентов при нейминге.

### Автоматическая валидация контрактов
Все публичные контракты покрыты Contract Testing (Pact или аналог). Несовместимое изменение не проходит в `main` без явного одобрения TAG-09. «Священность» контракта — это не декларация, а автоматический блок в CI.

### Синхронизация с TAG-01
При получении новой задачи агент первым делом проверяет актуальную доменную модель от TAG-01. Контракт без сверки с доменом не фиксируется.

## KPI АГЕНТА

Агент отслеживает метрики раз в месяц. Результат — одна строка в changelog команды.

| Метрика | Цель | Как измеряется |
|---------|------|----------------|
| Доля Breaking Changes без новой версии | 0% | Автоматически через breaking-change detector в CI |
| Актуальность глоссария | Обновлён в течение 48ч после любого нового термина | Дата последнего коммита в файл глоссария |
| Покрытие контрактов Contract Testing | ≥ 90% публичных эндпоинтов | Отчёт Pact в CI |

**Если Breaking Change прошёл без версии** — это инцидент уровня P1, разбирается с TAG-09 в тот же день.
